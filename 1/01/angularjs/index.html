https://angular.io

准备知识

  • Javascript
  • Node
  • TypeScript

快速开始

安装环境与创建一个项目

npm install -g @angular/cli
ng new my-app

默认文件结构

运行项目

cd my-app
ng serve --open

打开 locahost:4200 (默认端口 4200)

编译项目，从源码文件生成最终的应用程序文件

运行下面的命令

ng build

几秒后可以看到这个结果

注意，现在文件夹中得到一个新的子目录dist，稍后解释。

编辑第一个组件

Angular 自动创建了第一个组件，被称做 root compnent 的 name 为“app-root”的组件，代码在./src/app/app.component.ts

打开./src/app/app.component.ts， 修改里面的代码 title部分

export class AppComponent {
  title = 'My test app';   //app
}

然后打开关联的app.component.html文件，删掉多余的图片、链接，注意里面读取了title变量

<div style="text-align:center">
  <h1>
    Welcome to {{title}}!
  </h1>
</div>

编辑app.component.css，修改样式

h1 {
    color: #C00;
}

浏览器会自动刷新，html 中之前的 “app”已经被替换成了新的 title变量值，颜色也变啦。

注意到 src 文件夹中还有一个 app.component.spec.ts文件（和组件定义文件同名，但中间多了个spec），是单元测试用的，继续学习吧。

应用运行中加载的文件

angular 项目运行时的配置文件

运行 ng serve --open 命令的时候发生了什么？ 为什么会显示这个页面呢？这个页面来自哪个地方？页面上的文本/数据在哪里修改？

运行ng build时，“编译”什么？编译后又生成了什么？

首先明白：ng是一个命令行工具（cli）。

有过前端开发经验的人就知道，在一些使用了 webpack 的项目中，启动一个命令后，会读取一个配置文件，根据里面的参数来自动执行一些任务（例如文件压缩、合并、监听文件变动等）

同样，ng 命令运行后，读取的配置文件是项目根目录下的.angular-cli.json。

在编辑器里打开它，熟悉这个文件有助于理解 Angular 项目的文件加载流程。

我们现在只关注一部分和“项目结构”有关的配置信息，请看注释

// 与“项目”有关
"project": {
    "name": "angular-app"  // 项目名称
  },
  //项目下的 application 应用程序，目前只有一个
  "apps": [
    {
    	// 源文件目录，我们对项目进行开发就是在这里
      "root": "src", 
      // 源文件编译后的最终执行文件目录 （就是前面运行 ng build 生成的文件）
      "outDir": "dist",  
      //资源，包含多个文件/文件夹
      "assets": [
        "assets",
        "favicon.ico"
      ],
      // 首页（一个 html 页面）
      "index": "index.html",
      // 默认加载的的主程序文件（一般在这里面 import module）
      "main": "main.ts",
      "polyfills": "polyfills.ts",
      "test": "test.ts",
      "tsconfig": "tsconfig.app.json",
      "testTsconfig": "tsconfig.spec.json",
      "prefix": "app",
      "styles": [
        "styles.css"
      ],
      "scripts": [],
      "environmentSource": "environments/environment.ts",
      "environments": {
        "dev": "environments/environment.ts",
        "prod": "environments/environment.prod.ts"
      }
    }
  ],

入口文件 index.html

其中 index.html 是整个项目的“入口”。注意里面的<app-root></app-root>，稍后会讲到，这是一个“组件”(component)，也是项目的**root** componet。

这个根组件中，随着功能的增加，会逐渐增加很多的子组件。

运行 ng build 命令，cli 会自动把 js 和 css 文件添加到 html 代码中作为外部引入文件。

<body>
  <app-root></app-root>
</body>

src 目录，主要的源代码文件

查看自动生成的项目结构，其中的src目录就是我们项目的源码文件保存目录。

dist 源码编译后生成的最终应用程序文件

现在，源码文件夹src里面的很多 .ts文件（TypeScript）已经被编译成 浏览器/node 能识别的.js文件(Javascript)。

来看看最终生成的 index.html 和 src 里的 index.html 有啥不同

是的，这里面通过 ng build 命令后，引入了所有应用程序相关的 js 文件！！

<body>
  <app-root></app-root>
<script type="text/javascript" src="inline.bundle.js"></script><script type="text/javascript" src="polyfills.bundle.js"></script><script type="text/javascript" src="styles.bundle.js"></script><script type="text/javascript" src="vendor.bundle.js"></script><script type="text/javascript" src="main.bundle.js"></script></body>

默认加载的主程序文件 main.ts

The main entry point for your app Compiles the application with the JIT compiler and bootstraps the application’s root module (AppModule) to run in the browser

这一句，引入了app/app.module.ts文件

import { AppModule } from './app/app.module';

以及当前的运行环境是开发还是生产环境。

接上，AppModule

Defines AppModule, the root module that tells Angular how to assemble the application. Right now it declares only the AppComponent. Soon there will be more components to declare.

这里面又引入app.component.ts文件

import { AppComponent } from './app.component';

@NgModule({
  declarations: [
    AppComponent
  ],
  imports: [
    BrowserModule
  ],
  providers: [],
  bootstrap: [AppComponent]
})

终于，知道我们的 root component 是如何与 index.html 里的 产生关联的了！

app.component.ts文件中，定义了这个 root component，以及绑定的 html 和 css 文件，分别打开它们看看。

组件，后面学习

@Component({
  selector: 'app-root',
  templateUrl: './app.component.html',
  styleUrls: ['./app.component.css']
})
export class AppComponent {
  title = 'app';
}

assets 目录

这里可以放图片等静态资源文件，在 build 的时候会批量复制到应用程序目录中去

environments 目录

通过定义不同的目标运行环境参数，可以让程序在运行中，通过判断运行环境来执行不同的操作。

例如，在生产环境，可能需要加载 google 分析代码； 而在开发环境，需要显示一个“local”的提示

polyfilll.ts

Different browsers have different levels of support of the web standards. Polyfills help normalize those differences. You should be pretty safe with core-js and zone.js, but be sure to check out the Browser Support guide for more information.

style.css

Your global styles go here. Most of the time you’ll want to have local styles in your components for easier maintenance, but styles that affect all of your app need to be in a central place.

全局样式文件

test.ts

This is the main entry point for your unit tests. It has some custom configuration that might be unfamiliar, but it’s not something you’ll need to edit.

单元测试入口文件

tsconfig.{app|spec}.json

TypeScript compiler configuration for the Angular app (tsconfig.app.json) and for the unit tests (tsconfig.spec.json).

其他文件

e2e/ Inside e2e/ live the end-to-end tests. They shouldn’t be inside src/ because e2e tests are really a separate app that just so happens to test your main app. That’s also why they have their own tsconfig.e2e.json.

node_modules/ Node.js creates this folder and puts all third party modules listed in package.json inside of it.

.angular-cli.json Configuration for Angular CLI. In this file you can set several defaults and also configure what files are included when your project is built. Check out the official documentation if you want to know more.

.editorconfig Simple configuration for your editor to make sure everyone that uses your project has the same basic configuration. Most editors support an .editorconfig file. See http://editorconfig.org for more information.

.gitignore Git configuration to make sure autogenerated files are not commited to source control.

karma.conf.js Unit test configuration for the Karma test runner, used when running ng test.

package.json npm configuration listing the third party packages your project uses. You can also add your own custom scripts here.

protractor.conf.js End-to-end test configuration for Protractor, used when running ng e2e.

README.md Basic documentation for your project, pre-filled with CLI command information. Make sure to enhance it with project documentation so that anyone checking out the repo can build your app!

tsconfig.json TypeScript compiler configuration for your IDE to pick up and give you helpful tooling.

tslint.json Linting configuration for TSLint together with Codelyzer, used when running ng lint. Linting helps keep your code style consistent.
