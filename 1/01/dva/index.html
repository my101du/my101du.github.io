dva is a React and redux based, lightweight and elm-style framework, which supports side effects, hot module replacement, dynamic on demand, react-native, and SSR. It has been widely used in production environments.

elm 概念，通过 reducers, effects 和 subscriptions 组织 model

dva 经朋友推荐开始接触 dva ，从 1.x 版本开始使用，一直看着它发展到前不久的 2.x，我也基于这个工具开发了一套项目模版，它简化了 redux 的使用，并且在封装了 redux-saga 和 react-router，同时还可以包含 dva-loading 插件获取 loading 状态等。

在 redux 时代，当我需要新增一种跨页面全局数据的时候，我需要去项目的 reducers 目录定义一下这种数据命名和初始值，然后在 constans 目录中为更改这项数据的操作定义一种唯一的操作类型（type），再去 actions 目录定义一些方法，这些方法最后会得到更改后的数据和操作类型（type），最后再回到 reducers 中根据这个操作类型（type）把数据整合到 reducer 中…可以看到，我在编写 redux 这部分代码的时候需要频繁在 actions 、 constants 、 reducers 这几个目录间切换。

而使用 dva 就可以免除这些困扰了，我只需要一个 model 中就可以完成所有操作

dvi-cli

全套

npm install dva-cli -g
$ dva -v

$ dva new dva-quickstart
$ cd dva-quickstart
$ npm start

dva-core

对应 dva 里的“定义 Model”部分。dva 通过 model 的概念把一个领域的模型管理起来，包含同步更新 state 的 reducers，处理异步逻辑的 effects，订阅数据源的 subscriptions 。

例如 models/products.js

export default {
  namespace: 'products', //表示在全局 state 上的 key
  state: [], //是初始值，在这里是空数组
  reducers: { //等同于 redux 里的 reducer，接收 action，同步更新 state
    'delete'(state, { payload: id }) {
      return state.filter(item => item.id !== id);
    },
  },
};

载入它 （当然可以像 taro-dva 的 dva.js 一样，写 createApp 来初始化各种数据）

app.model(require('./models/products').default);

把 Component 和 model 两者 connect 起来

import { connect } from 'dva'; // taro 里直接用 from "@tarojsredux" dva 文档里说这两个一样？
import ProductList from '../components/ProductList';

const Products = ({ dispatch, products }) => {
  function handleDelete(id) {
    dispatch({
      type: 'products/delete',
      payload: id,
    });
  }
  return (
    <div>
      <h2>List of Products</h2>
      <ProductList onDelete={handleDelete} products={products} />
    </div>
  );
};

export default connect(({ products }) => ({
  products,
}))(Products);

models

redux 有 action/reducer 等一堆零散东西。简单来说，分成几块：

  1. 数据 （例如 post 博客）
  2. 修改数据的方法 （例如 edit()）
  3. 又一个方法： 调用修改数据的方法后，数据会怎么变化 （例如 saveEdit()）
  4. 这些方法分别叫什么名字 (例如 ‘edit’, ‘saveEdit’ 注意他们是字符串)

这些东西分散在 N 个目录和不同的文件下，虽然逻辑结构清晰，但编辑、管理起来实在是太痛苦了，改个方法要在几个文件里跳来跳去。随便增加一个动作就要添加几个文件。

还好， dva 可以把它们统一成“model”（模型）的概念，在一个文件里完成这一系列步骤。下面是一个 post 的 model

文件 MyApp/src/models/post.js

一个 model 关键的几个部分：

  1. namespace: 表示这个 model 的名称
  2. state：这个 model 的数据
  3. effects: model 可以有哪些“动作”（例如请求网络，获取数据）
  4. reducers: 对应某个 effect, 当完成动作后，对数据怎么操作
import Taro from '@tarojs/taro'
import action from "../utils/action";
import request from "../utils/request";
import delay from "../utils/delay";

export default {
  namespace: 'posts',

  state: {
    currentCategory: null,
    isFetching: false,
    updateTime: null,
    postsByCategories: {},
  },

  reducers: {
      postsByCategory(state, {payload}) {
        const {items, id} = payload;

        return {...state, postsByCategories: {...state.postsByCategories, [id]: items}}
    },
    //other reducer...
  },

  effects: {
    * search(_, {all, call, put}) {
      Taro.showLoading({
        title: '搜索中...',
      });
      try {
        let loadPro = yield put(action("load"));
        yield call(() => loadPro);
      } finally {
        Taro.hideLoading();
      }
    },
    
    * load({payload}, {all, call, put}) {
      let {data} = yield call(request, {
        url: 'https://easy-mock.com/mock/5b21d97f6b88957fa8a502f2/example/feed'
      });
      yield call(delay, 2000);//增加延迟测试效果
      yield put(action("save", {list: data}))
    },
    
    * fetchPostsByCategory({payload}, {all, call, put}) {
      const { id } = payload;

      // let {data}   
      let data = yield call(request, {
        url: SITE_URL + "/wp-json/wp/v2/posts?categories=" + id
      });

      // yield call(delay, 2000);//增加延迟测试效果
      yield put(action("postsByCategory", {items: data, id}))
    },
    // other effects...
  },
};

effect

在一个 effect 中，通过用 call() 来调用另外一个方法（例如，网络请求、获取文件内容等）

let data = yield call(request, {
    url: SITE_URL + "/wp-json/wp/v2/posts?categories=" + id
  });

当得到返回数据后，再调用put()方法触发一个 model 里对应的 reducer(例如这里“postsByCategory”，并带有参数)

yield put(action("postsByCategory", {items: data, id}))

reducer

在 effect 里使用 put()来触发某个 reducer. 在这里进行数据的后续处理。

改变 model 的 state 属性。

postsByCategory(state, {payload}) {
  const {items, id} = payload;

  return {...state, postsByCategories: {...state.postsByCategories, [id]: items}}
},

定义创建 dva app 实例的函数 createApp(opt)：关联 redux 的 stores 和 dva models

文件 MyApp/src/dva.js

这个函数：

  1. 接受一个opt参数，值包含有 models 列表
  2. 循环 opt.models, 使用 app.model(model)来向 dva app 注册 models
  3. 为 dva app 创建一个dispatch()方法，来自app._store.dispatch()
  4. 为 dva app 创建一个getStore()方法，返回app._store
  5. 最后返回一个 dva app 实例
import {create} from 'dva-core';
import {createLogger} from 'redux-logger';
import createLoading from 'dva-loading';

let app;
let store;
let dispatch;

function createApp(opt) {
  opt.onAction = [createLogger()];
  app = create(opt);
  app.use(createLoading({}));

  if (!global.registered) opt.models.forEach(model => app.model(model));
  global.registered = true;
  app.start();

  store = app._store;
  app.getStore = () => store;

  dispatch = store.dispatch;

  app.dispatch = dispatch;
  return app;
}

export default {
  createApp,
  getDispatch() {
    return app.dispatch;
  }
}

dispatch 是干什么的？

翻译来叫“分派”，在 Redux 里，其实就是一个函数，用来发送通知，驱动数据的变化。这个 dispatch 函数有两个参数

  1. 干什么
  2. 带

定义 action() 辅助方法 (非必需)：将 dispatch 的参数列表格式化

文件./utils/action.js

将按顺序列出的多个参数值（无索引名） [type, payload]，转换成一个有索引名的 js 对象 {type: type, payload: payload}

const action = (type, payload) => ({type, payload});

export default action

有了它，在写 dispatch() 调用时的参数时候，就可以简化为 (参数的值具体表示什么，等下在 model 里去看)

dispatch(action("post/get", {id: 123}));

否则就要完整写出 type, payload 名称.

dispatch({
    type: 'post/get',
    payload: {
        id: 123
    }
})

因为代码里会频繁调用 model 的方法，所以用这个 action()来简化代码
