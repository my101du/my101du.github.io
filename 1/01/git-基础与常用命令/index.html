Git 是一个非常好用的分布式版本控制系统

安装 Git 服务器

安装软件包

# Ubuntu
$ apt-get install git

# 或者 centos
$ yum -y install git

升级 git (vscode 的 remote ssh 开发，经常提示 git 1.8 版本低，需要2.0以上)

yum remove git
sudo yum -y install  https://centos7.iuscommunity.org/ius-release.rpm
sudo yum -y install  git2u-all

git --version

服务器创建 git 用户

在服务器上创建一个专用的 git 用户，用来运行 git 服务：

$ sudo adduser git

然后让创建的git用户不允许登录shell 通过编辑/etc/passwd文件完成。找到类似下面的一行：

git❌1001:1001:,,,:/home/git:/bin/bash

改为：

git❌1001:1001:,,,:/home/git:/usr/bin/git-shell

git用户可以正常通过ssh使用git，但无法登录shell，因为我们为git用户指定的git-shell每次一登录就自动退出

然后创建证书登录

收集所有需要登录的用户的公钥（id_rsa.pub），导入到 /home/git/.ssh/authorized_keys文件里

在每台设备上

ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

# 一直回车，在当前用户目录下生成 .ssh/id_rsa.pub 文件

# 复制文件内容，粘贴到 github/coding/git服务器 的公钥管理

使用 nginx 搭建 http 访问和验证的 git server

默认用 ssh 协议 user@server:/path/to/repo 来 clone /push 代码

但是也可以搭建一个类似 github 支持 http 协议的 git server.(例如 php composer 仓库就需要 http 类型)

https://beginor.github.io/2016/03/12/http-git-server-on-nginx.html

使用 git-http-backend 搭建 git 服务的原理都是类似的， 主要是利用 web 服务器 (apache/nginx) 进行用户认证， 并将用户信息传递给 CGI 程序 git-http-backend ， 从而实现通过 http 完成 git 操作。

Nginx — git-http-backend —- git repo

安装 fcgiwrap

Nginx 作为 Web 服务器，本身是不能执行外部 CGI 脚本的，需要通过 fcgiwrap 来中转，就像使用 php-fpm 来执行 PHP 脚本一样。

apt-get install git-core nginx fcgiwrap

nginx

添加一个虚拟目录 /git/ ， 通过访问 /git/xxx.git 的形式来访问服务器上的 xxx.git 代码库

修改一下 nginx 默认网站的配置文件 /etc/nginx/sites-available/default

注意： 这个 location ~ /git(/.*) 无效。。。也会报错 not found, 改成 / 才行 GIT_PROJECT_ROOT 不支持“虚拟机映射的宿主机目录”，后面会提示 not found

例子：root 目录是 /var/html/git 然后 location 是 /. 然后项目目录放在 /var/www/git/test.git （注意权限问题）

# 配置以 /git 开始的虚拟目录
location ~ /git(/.*) {
    # 使用 Basic 认证
    auth_basic "Restricted";
    # 认证的用户文件
    auth_basic_user_file /etc/nginx/passwd;
    # FastCGI 参数
    fastcgi_pass  unix:/var/run/fcgiwrap.socket;
    fastcgi_param SCRIPT_FILENAME /usr/lib/git-core/git-http-backend;
    fastcgi_param GIT_HTTP_EXPORT_ALL "";
    # git 库在服务器上的跟目录
    fastcgi_param GIT_PROJECT_ROOT    /var/git-repos;
    fastcgi_param PATH_INFO           $1;
    # 将认证用户信息传递给 fastcgi 程序
    fastcgi_param REMOTE_USER $remote_user;
    # 包涵默认的 fastcgi 参数；
    include       fastcgi_params;
    # 将允许客户端 post 的最大值调整为 100 兆
    max_client_body_size 100M;
}

合并分支 merge & rebase

两种方式： merge 和 rebase

例如，有一个分支 bugfix 从 master 分叉出来

merge 方式

根据 master 分支在 bugfix 分支变化期间，自身是否有变化，又分两种情况：

master 分支有变化：需要汇总后，生成一个新提交

把两个分支修改的内容汇合起来！生成一个提交. master分支的HEAD会移动到该新的提交上

master 如果没有变化：fast-forward（快进合并），直接移动 HEAD 位置

假设在 bugfix 的多次修改中，master 分支都没有变化, 这时 通过把master分支的位置移动到bugfix的最新分支上，Git 就会合并

当然，如果你想强行即使满足 fast-forward 也不要快速合并，而是生成新的提交并合并。可以使用参数non fast-forward，那么就是下面的图了

# 1: do something in develop branch, and commit the changes
git checkout develop
<in develop> vi README.md
<in develop> git add .
<in develop> git commit -m "modify readme"

# 2. switch to master
git checkout master

# 3. merge the changes from develop to master
<in master> git merge develop

rebase 方式

但是，上面用 merge 的方式有点“脏”，因为最终会看到多个分支 （master, bugfix）的变化历史记录

使用 rebase, 可以把历史记录成一条线，更加整洁

首先，rebase bugfix 分支到master分支, bugfix分支的历史记录会添加在master分支的后面. 因为 rebase 之后，master的HEAD位置不变, 要合并master分支和bugfix分支，即是将master的HEAD移动到bugfix的HEAD这里

如果在这个过程中(提交 X 和 Y),会提示冲突, 这时要修改各自提交时发生冲突的部分

最终

对比两种方式

  • merge: 保持修改内容的历史记录，但是历史记录会很复杂
  • rebase: 历史记录简单，是在原有提交的基础上将差异内容反映进去。因此，可能导致原本的提交内容无法正常运行

根据开发团队的需要分别使用merge和rebase。 例如，想简化历史记录

在topic分支中更新merge分支的最新代码，请使用rebase。 向merge分支导入topic分支的话，先使用rebase，再使用merge。

解决冲突

模拟两个人同时修改一个文件：

假设 A 和 B 分别从 master 分支创建了 feature-1 和 feature-2 分支

A 在 feature-1 上修改文本 readme.txt

1. git add
2. git commit

merge 合并

在 master 上合并(由于 master 分支并没有变化，会执行 fast-forward（快进）合并)，一切正常

git merge feature-1

然后 B 修改同一个文件，内容

1. git add
2. git push

回到 master 运行合并, 报了冲突错误

git merge feature-2

Auto-merging myfile.txt
CONFLICT (content): Merge conflict in myfile.txt
Automatic merge failed; fix conflicts and then commit the result.

这是因为两个分支同时改动了文件的第二行，内容不一致，需要手动解决冲突

1. git add
<<<<<<< HEAD
2. git commit
=======
3. git push
>>>>>>> feature-2

用文本编辑器改成

1. git add
2. git commit
3. git push

重新提交

git add myfile.txt
$ git commit -m "合并分支"

因为在这次合并中修改了冲突部分，所以会重新创建合并修改的提交记录。这样，master的HEAD就移动到这里了。这种合并不是fast-forward合并，而是non fast-forward合并。

假如改成在 feature-2 上对 master 执行 rebase 合并呢？

  1. 先暂时取消刚才的合并操作
git reset --hard HEAD~

情况是这样

  1. 切到 feature-2 分支，B 这时已经知道 master 分支已经把 A 提交的 feature-2 分支给合并进去了

  2. 于是对 master 执行 rebase （有点类似 pull?）

git checkout feature-2

git rebase master

# First, rewinding head .....
# <stdin>:13: new blank line at EO....
# CONFLICT (content): Merge conflict in myfile.txt.....

和之前 merge时的操作相同，修改发生冲突的部分(略)

  • 此时，修改冲突发生在 feature-2 分支上，而不是前面在 master 上去合并它（发生位置刚好是个反的）
  1. commit 提交解决冲突

rebase的时候，修改冲突后的提交不是使用commit命令，而是执行rebase命令指定 –continue选项。若要取消rebase，指定 –abort选项

git add myfile.txt

git rebase --continue
  1. 回到 master 进行 fast-forward 合并

由于刚才在 feature-2 上对 master 进行 rebase，然后已经把 feature-2 的冲突解决了。

这时 master 分支可以进行fast-forward 合并

$ git checkout master
Switched to branch 'master'

$ git merge feature-2

Updating 8f7aa27..96a0ff0
Fast-forward
 myfile.txt |    1 +
 1 files changed, 1 insertions(+), 0 deletions(-)

注意对比前面在 master 分支上解决冲突的历史记录，当前的情景干净多了

rebase 适合什么场景

按我自己的理解

merge 会记录所有历史记录，所以比较适合在 master 分支上对其他分支进行合并，比较看重整个项目中别人的工作情况。

rebase 一般发生在非 master 分支上，是开发者自己把 master 上已经产生的变化合并到本地，不太关心别人做了什么具体的，只关心自己当前的代码是不是保持最新的

非常适合一个开发者的下面的场景

  1. 正在开发一个新功能，分支 feature-1
  2. 突然被告知线上 master 有个 bug 要处理
  3. 如果是分配给自己处理，自己赶紧把工作保存，再去基于 master 新建个分支 bugfix-1
  4. 如果是别人处理，那就只要知道就可以了
  5. 修改完 bug，合并到 master
  6. 继续回来开发 feature-1, 因为这时代码已经不是最新的了，所以先 rebase 下 master 分支，因为当前分支主要是开发新功能，只要能和 master 保持同步就好，没有必要merge bugfix-1 分支去保存多余的信息

使用git rebase是有条件的，你的本地仓库要“足够干净”。可以用git status命令查看当前改动：

本地没有任何未提交的改动，这是最“干净”的。 或者 本地只有新增文件未提交，没有改动文件

merge 分支到主分支，不保留历史记录

默认情况下。 如果在 develop 分支上，有了两个修订版，再合并回 master 后，两次修改的记录都会保留。

git commit -m "initial"

git checkout -b develop

# modify 1  -> add  -> commit
# modify 2  -> add  -> commit

git checkout master

git merge develop

current modify 2 modify 1 initial

4个区与三个步骤

工作区(Working Area) git add .把所有文件放入暂存区； 暂存区(Stage) git commit把所有文件从暂存区提交进本地仓库； 本地仓库(Local Repository) git push把所有文件从本地仓库推送进远程仓库。 远程仓库(Remote Repository)

5种状态

未修改(Origin) 已修改(Modified) 已暂存(Staged) 已提交(Committed) 已推送(Pushed)

检查修改

git diff # 检查工作区和暂存区之间的差异（运行 git add . 后，本地又有修改）

git diff --cached # 检查暂存区和本地仓库之间的差异 （已暂存，未提交, 即上次 git commit 后，又 git add .）

git diff master origin/master # 检查本地仓库和远程仓库区别 (已提交，未推送到远端 即上次 git push 后，本地又 git commit)

已修改，未暂存的回退

如果我们只是在编辑器里修改了文件，但还没有执行git add .，这时候我们的文件还在工作区，并没有进入暂存区

git checkout .

#或者
git reset --hard

git add 添加 多余文件

有的时候 git add . 把不需要的文件给 add 进去了. (还没有执行git commit -m “comment”)可以通过下面的方法撤销

git reset # 把修改退回到了git add .之前的状态(处于已修改未暂存状态)
git checkout . # 继续退回未修改状态

#或者替代的一句：
git reset --hard

# 退回某个文件
git status #先看一下add 中的文件
git reset HEAD #如果后面什么都不跟的话 就是上一次add 里面的全部撤销 
git reset HEAD XXX/XXX/XXX.java #对某个文件进行撤销

git commit 错误

如果不小心 弄错了 git add后 ， 又 git commit 了(代码已经进入了你的本地仓库)

# 相当于从远程仓库把代码取回
git reset --hard origin/master

如果要恢复到倒数第二次提交

git log #查看节点 

# 最新的一次提交（当前）
commit xxxxxxxxxxxxxxxxxxxxxxxxx3 
Merge: 
Author: 
Date:

# 倒数第二次提交
commit xxxxxxxxxxxxxxxxxxxxxxxxx2 

# 倒数第三次提交
commit xxxxxxxxxxxxxxxxxxxxxxxxx1 

# 保持代码不变, 还是最近一次提交的状态
git reset xxxxxxxxxxxxxxxxxxxxxxxxx2

# 加上 --hard 参数，代码也恢复到倒数第二次提交的状态
git reset --hard commit_id

push 以后(代码已经进入远程仓库)

先恢复本地仓库，再强制push到远程仓库

git reset --hard HEAD^
git push -f

还原已经提交的修改 此次操作之前和之后的commit和history都会保留，并且把这次撤销作为一次最新的提交

git revert是 提交一个新的版本，将需要revert的版本的内容再反向修改回去，版本会递增，不影响之前提交的内容。

git revert HEAD #撤销前一次 commit 
git revert HEAD^ #撤销前前一次 commit 
git revert commit-id #(撤销指定的版本，撤销也会作为一次提交进行保存）

# 默认会进入 master 分支
$ git branch

# 如果要进入 develop 分支开发，需要指定远程库的 origin/develop 分支，把它 checkout 到本地
# 拉取本地不存在的远程分支
$ git checkout -b develop origin/develop

# git clone的时候，所有本地分支会自动与远程主机的同名分支之间建立追踪关系，所以无需手动建立跟踪关系
# 不对
# 默认 clone 的是这个仓库的 master 分支。如果最新的代码不在 master 分支上，该如何拿到呢？

git branch -r #查看远程分支
git branch -a #查看所有分支
git checkout origin/daily/1.4.1  # 直接用这个完整的

# 原来写的这个方法更多的是：设置已有的本地分支跟踪一个刚刚拉取下来的远程分支，或者想要修改正在跟踪的上游分支。

通过添加远程主机的方式

$ mkdir myproject
$ cd myproject

#把本地一个目录初始化为 git 管理的项目(会自动创建一个 .git 隐藏目录)
$ git init

#把远程主机上的版本库添加进来，别名为 origin 
$ git remote add origin https://git.github.com/username/myproject.git   

#对比一下本地和远程主机，显示如下
$ git remote -v
origin	https://github.com/username/myproject.git (fetch)
origin	https://github.com/username/myproject.git (push)

#最后把远程主机的所有更新取回本地，也可以只取回特定分支如 git fetch origin master
$ git fetch origin

# 查看远程主机的分支 ， -a 参数查看所有分支(包括本地)
$ git branch -r 

# 可以在取回远程主机的更新后，在本地仓库创建一个新分支(最好让本地仓库和远程仓库的分支名保持一样)
$ git checkout -b develop origin/develop #会自动切换到这个新的分支上

# 合并远程分支内容到本地当前工作分支
$ git branch -a # 先查看当前工作在哪个分支（前面有星号`*`的）
* develop
  remotes/origin/develop
  remotes/origin/master
$ git merge origin/develop # 然后可以在当前工作分支上，把远程分支合并进当前工作分支
 
# 上面的先 fetch 取回更新，然后与本地指定分支 merge 合并，可以使用 pull 代替，pull=fetch+merge
$ git pull origin develop  # 默认是与当前分支(初始化的时候就是 master )合并，如果用 develop:master 表示取回远程主机的 develop 分支，与本地 master 分支(非当前分支)合并

# 建立本地分支与远程分支之间的追踪关系（pull 的时候用到）
$ git branch --set-upstream master origin/next # 手动建立追踪关系

本地修改文件后，提交到本地分支，最后推送到远程分支

$ vi README.md
$ git add ./README.md
$ git commit -m "modify readme.md"

# 推送到远程分支
# git push <远程主机名> <本地分支名>:<远程分支名> ，如果本地分支和远程分支有追踪关系，可以省略远程分支名(甚至如果当前分支有追踪关系，可以直接省略本地分支名)
$ git push origin develop

# 如果当前分支与多个主机存在追踪关系，可以使用 -u 指定默认主机
$ git push -u origin develop

#如果远程主机的版本比本地版本更新，推送时Git会报错，要求先在本地做git pull合并差异，然后再推送到远程主机

使用 statsh 来暂存，将暂时不想提交的内容临时保存

较常用的几个命令：

  • git stash
  • git stash list
  • git stash clear
  • git stash apply

git stash 将当前所有修改项(未提交的)暂存，压栈。此时代码回到你上一次的提交，用git status可查看状态。

  • git stash list将列出所有暂存项。
  • git stash clear 清除所有暂存项。
  • git stash apply 将暂存的修改重新应用，使用git status可以看到以前暂存的修改又回来了

这几个命令常用来干几件事情。当修改项还不确定要提交，又要将代码更新或则回去改bug，此时可以用git stash将修改项暂存，然后想干嘛干嘛，干完之后git stash apply，以前的修改项又回来了，可以继续完成修改。
