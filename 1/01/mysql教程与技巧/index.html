在 CentOS 7 下安装

安装服务 & 启动

# MariaDB
sudo yum install mariadb-server mariadb
sudo systemctl start mariadb

# MySQL
wget http://repo.mysql.com/mysql-community-release-el7-5.noarch.rpm
sudo rpm -ivh mysql-community-release-el7-5.noarch.rpm
yum update

sudo yum install mysql-server

# 查看默认生成的root密码，后面用到
sudo grep 'temporary password' /var/log/mysqld.log

后面就一样的了。

初始化数据库

sudo mysql_secure_installation

一开始默认 root 密码是空的，直接回车，然后输入新密码

开机启动

sudo systemctl enable mariadb

命令

使用 mysql client 工具连接到服务器

# 本地默认 localhost  3306 端口
mysql -uroot -p

# 假设本机 ip 是 192.168.0.7
# 远程机器别的端口（在其他机器上要开放 developer@192.168.0.7）
# -h 指定主机名或 ip， --port= 指定端口
mysql -udeveloper -p -h 192.168.0.8 --port=3307

用户

# 创建用户
CREATE USER 'username'@'host' IDENTIFIED BY 'password';

# 查看用户拥有的 权限
show grants for test_user@'localhost';

权限

GRANT ALL PRIVILEGES ON *.* TO ''@'localhost' IDENTIFIED BY 'PASSWORD' WITH GRANT OPTION;

进入 MySQL 命令行后执行 sql 文件

如果 sql 文件不大

可以打开文件，在第一行添加

use database_name;

如果正运行mysql，或sql文件编辑不方便，可以使用source或.命令执行SQL脚本文件：

# 先准备好 sql 文件，注意使用 use 来标记当前数据库
$ vi filename.sql

# 然后进入MySQL
$ mysql -uroot -p

# 执行
mysql> source filename.sql

# 或者
mysql> \. filename

导入导出

注意以下导入导出命令中，箭头的方向。

从某个备份文件（.sql）导入

# 使用 mysql 命令
$mysql -uroot -p datbase_name < /backup/restore_xxxxx_xx_xx.sql

导出

# 使用 mysqldump 命令

# 导出整个库
$mysqldump -uroot -p database_name > /backup/restore_xxxx_xx_xx.sql

#导出库里面的某个表
$mysqldump -uroot -p database_name table_name > /backup/restore_table_xxxx_xx_xx.sql

将 SQL 语句的查询结果导出成 csv 文件

SELECT order_id,product_name,qty
FROM orders
WHERE foo = 'bar'
INTO OUTFILE '/var/lib/mysql-files/orders.csv'
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n';

复制表（结构、索引、数据）

To copy with indexes and triggers do these 2 queries:

CREATE TABLE newtable LIKE oldtable;

数据 data

To copy just structure and data use this one: （没有索引）

CREATE TABLE tbl_new AS SELECT * FROM tbl_old;

可以用下面的 DIV 来实现自动切割（先转换到时间戳）

SELECT
    `update_date`,
    `status`,
    count(STATUS)
FROM `group_by_range` b 
WHERE `status`<>''
GROUP BY 
UNIX_TIMESTAMP(`update_date`) DIV 3600, `status`;

一些常见问题

远程连接 mysql 提示连不上的问题

已经设置了 user_name@% 允许远程连接，已经 grant 了权限，还是连不上远程的服务器

检查远程服务器的 my.cnf，把 bind_address=127.0.0.1 注释掉。

坑爹的是，有时候这个文件可能会分布在不同的目录，还会“叠加”几个配置文件为一个完整的。

所以如果其中某个里面有这行，很容易被忽略

查看 mysql 调用了哪个（哪些）配置文件，在 MySQL 5.7 里可以用下面的命令看到

mysqladmin --help | grep -A1 'Default options'

MySQL的字段是不区分大小写的！

因此不管是搜索，还是批量处理数据，都要小心这点。 字符串查询是否？

修改 root 密码（5.6为例）

Linux

# 先用当前root密码登录 MySQL，在 MySQL> 提示符下操作

use mysql;
update user set password=password('yourpass') where user='root';
flush privileges;

Windows

打开一个DOS窗口，输入：net stop mysql56，以停止MySQL服务； 切换到MySQL的bin目录，如： cd C:\Program Files\MySQL\MySQL Server 5.6\bin 在DOS窗口中输入：mysqld –skip-grant-tables，回车 再打开一个DOS窗口，将切换到MySQL的bin目录（参考上面）； 输入：mysql -u root，回车以登录 登录成功后，使用update语句更改密码：update mysql.user set password=password(‘yourpass’) where user='root’ and host='localhost’; 然后重新装载权限：flush privileges; 输入exit;以退出到DOS窗口； 在资源管理器中杀掉mysqld进程，然后在DOS窗口中输入：net start mysql56，以重新启动mysql服务。 关闭已经开启的两个DOS窗口；打开一个MySQL 5.6 Command Line Client窗口，即可使用新密码登录。

If you forget your root MySQL password, it can be reset.

恢复数据库的时候，提示"Gone Away"的问题

在还原一个从服务器上下载的 sql 文件到本地开发环境的时候，提示如下错误。

mysql -uroot -p database_name < ~/backup/database_bak.sql
#ERROR 2006 (HY000) at line 1211: MySQL server has gone away

通过修改 my.cnf文件，调整下面这两个参数

vi /etc/my.cnf

#修改参数
[mysqld]
wait_timeout=600
max_allowed_packet=100M

注意要检查 my.cnf 文件里的参数，如果它里面定义了 max_allowed_package, 在命令行中指定参数，其优先级不如这个配置文件里定义的。

如果服务器只启用了 MyISAM引擎的话，应该重新还原就正常了。如果启用了 InnoDB 引擎，可能还会有如下错误

ERROR 1118 (42000) at line 1211: The size of BLOB/TEXT data inserted in one transaction is greater than 10% of redo log size. Increase the redo log size using innodb_log_file_size.

修改my.cnf的参数即可。

innodb_log_file_size=15M

MySQL 5.7 的sql_mode导致 INSERT SELECT GROUP BY 失败

以前有个计划任务在运行sql语句，一直运行正常，升级到 5.7 后，突然发现表空了。检查代码没问题。然后直接在 MySQL 里执行语句，才发现报错：

Getting this SQL Error: GROUP BY incompatible with sql_mode=only_full_group_by

解决方法是修改 my.cnf, 修改下面的内容

#sql_mode=xxxxx
sql_mode=STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION

MySQL 5.7 高版本备份出来的 sql 在低版本 MySQL 里还原提示编码错误

提示

1273 – Unknown collation: ‘utf8mb4_unicode_520_ci’

https://stackoverflow.com/questions/42385099/1273-unknown-collation-utf8mb4-unicode-520-ci

将全部编码改成 5.5 支持的就好

sed -i 's/utf8mb4_unicode_520_ci/utf8mb4_unicode_ci/g' file.sql

MySQL 字段长度限制 1000 字节的问题

在一次使用 Laravel 安装一个包运行 migration 时，数据库报错：

mysql：Specified key was too long; max key leng

因为 MyISAM 引擎对索引长度最大只支持 1000 bytes.

如果出现几个 varchar(255)，并且由于 utf8mb4 编码，一个字符占 3 个字节

255x3 + 255x3

就超过 1000 了。。。。

因此要把数据表转成 innoDB 引擎 （或者更改字符集、修改字段长度，不过这两个往往受限于应用系统要求没那么容易处理）

但还是

[PDOException] SQLSTATE[42000]: Syntax error or access violation: 1071 Specified key was too long; max key length is 767 bytes

得出的结论是：对于创建innodb的组合索引，如果各个列中的长度不超过767，则不再计算所有列的总长度，如果有超过767的，则给出报警，索引最后创建成功，但是对于超过767字节的列取前缀索引；对于innodb的单列索引，超过767的，给出warning，最终索引创建成功，取前缀索引（取前255字节）。

http://www.cnblogs.com/littleatp/p/4612896.html

解决

1 使用innodb引擎； 2 启用innodb_large_prefix选项，将约束项扩展至3072byte； 3 重新创建数据库；

my.cnf配置：

default-storage-engine=INNODB
innodb_large_prefix=on

如果不需要使用 特殊汉字 emoj，还可以把 utf8mb4 改成 utf8

一般情况下不建议使用这么长的索引，对性能有一定影响；

参考文档： https://dev.mysql.com/doc/refman/5.5/en/innodb-restrictions.html

若不想修改 mysql 配置，就可以修改 laravel 应用的 app\Providers\AppServiceProvider.php 文件

use Illuminate\Support\Facades\Schema;

public function boot()
{
    Schema::defaultStringLength(191);
}

或者根据需要，把 sql 文件里的 varchar(255) 改小，如 varchar(100)

使用utf8mb4字符集,长度超过255的varchar字段加索引会导致此问题 – 方案1(推荐) 情况允许,设置varchar字段长度为<=191的值 – 方案2 设置mysql参数： [mysqld] innodb_file_format = Barracuda #innodb_file_per_table = 1 innodb_large_prefix

在建表语句指定ROW_FORMAT： ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 ROW_FORMAT=DYNAMIC;

密码强度审计

如果使用弱密码，在安全性方面，会存在非常严重的安全隐患。

在 5.6 及以上版本中 ，提供了一个插件Password Validation Plugin用于要求用户使用强密码.

插件对应的库对象文件需在配置选项plugin_dir指定的目录中。可使用–plugin-load=validate_password.so，在Server启动时载入插件，或者将plugin-load=validate_password.so写入配置文件。也可以通过如下语句在Server运行时载入插件(会注册进mysql.plugins表)

运行时载入插件

mysql> INSTALL PLUGIN validate_password SONAME 'validate_password.so';

在配置文件中添加

[mysqld]

plugin-load=validate_password.so
validate_password_policy=2
validate-password=FORCE_PLUS_PERMANENT

插件关闭

关闭插件很简单，在MySQL配置文件(Centos系统下是/etc/my.conf)里面[mysqld]选项下面添加下面一条语句即可。

[mysqld]

validate_password=off

记得配置后要重启MySQL,在Shell下面运行下面两条语句

$ service mysqld stop
$ service mysqld start

验证插件

通过命令SHOW PLUGINS进行观察，应该观察到插件已启用。

测试插件

设置简单的密码，则MySQL数据库会报类似如下的错误

mysql> SET PASSWORD = PASSWORD('abc');
ERROR 1819 (HY000): Your password does not satisfy the current policy requirements

设置复杂密码，则可以成功修改。

mysql> SET PASSWORD = '*0D3CED9BEC10A777AEC23CCC353A8C08A633045E';

但是突然发现内存不够， 导致经常数据库停止运行。 于是降级到 5.5

但是在5.6上导出的备份，导入 5.5 的时候，不少表都丢失了（用的宝塔面板）

更换默认的 data 目录

下面是直接更换， 还可以做一个软连接。

# 检查mysql数据库存放目录
mysql -u root -p

# 查看sql存储路径
show variables like '%dir%';  

# 停止mysql服务
service mysqld stop

# 创建新的数据库存放目录
mkdir /data/mysql

# 移动/复制之前存放数据库目录文件，到新的数据库存放目录位置
cp -R /usr/local/mysql/data/* /data/mysql/    #或mv /usr/local/mysql/data/* /data/mysql

#修改mysql数据库目录权限以及配置文件
chown mysql:mysql -R /data/mysql/

vim /etc/my.cnf
#datadir=/data/mysql （制定为新的数据存放目录）

vim /etc/init.d/mysqld
#datadir=/data/mysql

#启动数据库服务
service mysqld start
