开始

是一个快速开发环境，能够使用 app, 服务，数据平台等, 并调用各种数据源，为商业需求创建应用。 powerapps 提供了很多的商业逻辑，以及与 flow 结合来提供流程自动化。

面向应用应用创建者与开发者

有两种类型的应用：

  • “画布”（PowerApps Studio 生成画布应用的应用设计器） 通过数据生成应用
  • “模型驱动”（“应用设计器”定义网站地图 模型驱动 的应用）

开发者可以编写代码以扩展业务应用的创建和自定义，使用代码创建数据和元数据，使用 Azure Functions、插件和工作流扩展应用服务器端逻辑，使用 JavaScript 应用客户端逻辑，使用虚拟实体和 webhook 与外部数据集成，构建自定义连接器，并将应用嵌入到网站体验中以创建集成解决方案。

delegation 委托：将数据处理委托给数据源，而不是在应用本地来处理

PowerApps 提供了大量的函数来进行 Filter、Sort、Add Column 等，来操作后端的数据库

我们要尽可能地让设备上的 app 和数据库之间的数据包越少越好，以便节约带宽、内存，提高 app 的性能

Delegation 会将数据处理委托给数据源，而不是在应用本地来处理

因为并不是所有可以在PowerApps公式中表达的内容都可以委托给每个数据源。PowerApps语言模仿了Excel的公式语言，设计了完整的、即时访问内存中的完整工作簿的功能，以及各种数字和文本操作函数。因此，PowerApps语言比大多数数据源所能支持的要丰富得多，包括强大的数据库引擎(如SQL Server)。

处理大型数据集需要使用可以委托的数据源和公式。

这是保持你的应用程序运行良好并确保用户能够访问他们需要的所有信息的唯一方法。注意委托警告，它指出不可能进行委托的地方。如果您使用的是小数据集(少于500条记录)，您可以使用任何数据源和公式，因为如果公式不能委托，应用程序可以在本地处理数据。

关于委托警告

如果数据源中的数据超过500条记录，且无法委托某个函数，PowerApps可能无法检索所有数据，应用程序可能会得到错误的结果。委托警告帮助您管理应用程序，使其具有正确的结果。

PowerApps在创建包含不能委托的内容的公式时提供警告(黄色三角形)。

委托警告只出现在对可委托数据源进行操作的公式上。如果您没有看到警告，并且您认为您的公式没有被正确地委托，请对照本主题前面的可委托数据源列表检查数据源的类型。

可委托的数据源

如果数据源支持委托，则其连接器文档将概述该支持， 例如

Common Data Service (CDS) for Apps SharePoint SQL Server

另外，导入的 excel 数据表，集合，以及保存在上下文变量中的表不需要委托，它们在内存中进行。

可委托的函数

由于每个数据源都是不同的，并不是所有数据源都支持所有这些元素。检查特定公式中的委托警告

Filter functions

Filter, Search, and LookUp can be delegated

  • And (including &&), Or (including ||), Not (including !)
  • In
  • =, <>, >=, <=, >, <
  • +, -
  • TrimEnds
  • IsBlank
  • StartsWith
  • Constant values that are the same across all records, such as control properties and global and context variables. (暂时不明白)

您还可以使用公式中对所有记录求值为常量的部分。例如，Left(Language()， 2)不依赖于记录的任何列，因此，对所有记录返回相同的值。它实际上是一个常数。上下文变量、集合和信号的使用可能不是常量，因此将阻止 Filter 和 Loopup 被委托

还有其他

  • If
  • *, /, Mod
  • Concatenate (including &)
  • ExactIn
  • String manipulation functions: Lower, Upper, Left, Mid, Len, …
  • Signals: Location, Acceleration, Compass, …
  • Volatiles: Now, Today, Rand, …
  • Collections

Sorting functions

Sort and SortByColumns can be delegated.

Sort 公式不能包含其他操作和函数，只能是列

Aggregate 汇总函数

Sum, Average, Min, and Max 等 (查看 https://docs.microsoft.com/en-us/powerapps/maker/canvas-apps/delegation-list)

但是这些计算函数不能被委托

例如 CountRows, CountA, and Count 以及其他统计函数 StdevP and VarP 不能被委托

不支持委托的函数

  • Table shaping: AddColumns, DropColumns, ShowColumns, …
  • First, FirstN, Last, LastN
  • Choices
  • Concat
  • Collect, ClearCollect
  • CountIf, RemoveIf, UpdateIf
  • GroupBy, Ungroup

一种常见的模式是使用AddColumns和LookUp将信息从一个表合并到另一个表中，这在数据库术语中通常称为连接

AddColumns( Products, "Supplier Name", LookUp( Suppliers, Suppliers.ID = Product.SupplierID ).Name )

即使 Products 和 Suppliers 都是可委托数据源，Lookup 也是可委托函数，但是 AddColumns 是不可委托的。整个公式的结果将局限在 Product 数据源的第一部分

因为查找函数及其数据源是可委托的，所以可以在数据源中的任何位置找到与供应商的匹配，即使它很大。一个潜在的缺点是，对于产品中的第一个记录，查找将对数据源进行单独的调用，从而在网络上引起大量的讨论。如果供应商足够小并且不经常更改，那么您可以在应用程序启动时(使用打开屏幕上的OnVisible)通过对方付费电话将数据源缓存到应用程序中，然后对其进行查找。

on-delegable limits 限制

非委托公式都运行在本地。所有的数据必须首先被带到设备上，这可能需要通过网络检索大量的数据。这可能需要时间，给人的印象是你的应用程序很慢，或者可能会崩溃。

为了避免这种情况，PowerApps对本地可处理的数据量进行了限制:默认情况下为500条记录。

考虑一种情况，需要过滤一批数据，它的“选择公式”不能在100万记录的数据源上进行委托，因为过滤是在本地完成的，只会扫描前500行数据。因此筛选器无法找到后面的数据。

还有聚合函数也会引起混淆。取同一百万记录数据源的一列的平均值。Average还不能被委托，所以只有前500条记录被平均。如果你不小心，部分答案可能会被你的应用程序的用户误解为完整的答案。

修改限制 修改 App settings ， Experimental features， Data row limit for non-delegable queries 如果改成 2000（相应地性能会有所下降）

为了确保您的应用程序可以扩展到大数据集，请将该设置设置为1。任何不能委托的内容都会返回一条记录，这在测试应用程序时应该很容易检测到。这有助于避免在将概念验证应用程序投入生产时出现意外

案例

在一个连接了 dbo.fruit SQLServer 数据源的应用中。一开始用

Filter('[dbo].[Fruit]', TextSearchBox1.Text in Name)

如果用 “apple” 做关键词，会找到

apples Pineapple Crab apples

等。速度还挺快，即使原来的数据表里有几百万条记录

然后修改，让搜索结果，必须与搜索关键词 apple 完全相同（不能在中间）

Filter('[dbo].[Fruit]', Left(FruitName, Len(TextSearchBox1.Text in Name)) = TextSearchBox1.Text)

这时界面上出现一个三角形黄色警告，以及公式输入框里波浪线标记了问题。

数据源是可委托的，Filter 也是委托函数，但是 Len()不能委托给任何数据源

如果表的数据小于500条，则正常，会将所有记录导入设备，在本地用 Filter. 否则大于500条，公式的结果可能不准确（拿不到500条后面的记录）
