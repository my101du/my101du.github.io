接着完善 PrivateRoute，实际上是一个 HOC, 设置一个 <Route>组件的render属性，判断当用户已经通过验证时，显示传入的组件，如果没有则 redirect 到登录的路由 /login

Redirect的to属性里，传入了一些数据到 /login（ props.location）便于登录完后返回这个页面，在后面用const { from } = this.props.location.state来取值

const PrivateRoute = ({ component: Component, ...rest }) => (
  <Route
    {...rest}
    render={props =>
      fakeAuth.isAuthenticated ? (
        <Component {...props} />
      ) : (
        <Redirect
          to={% raw %}{{
            pathname: "/login",
            state: { 
                from: props.location 
            }
          }}{% endraw %}
        />
      )
    }
  />
);

注意 不能用 class PrivateRoute extends Component {},在内部render(){ return ( <Route>) }的方式，会报错“函数内不能 return Component”, 因为 const PrivateRoute 是个“无状态组件”。不能用 return 的方式

如果用了 mobx, 那么 fakeAuth 就不能用了。

这段代码要改下，把 <Route> 放到一个 React 组件里面去，(上面那个无状态组件不支持 @inject这种指令的)

最好是把 this.props.component 命名为 Children 以免和 React 的 Component 类重名了。

@inject('authStore')
@observer
class PrivateRoute extends Component {
  render() {
    const { component:Children } = this.props

    return (<Route
      render={props =>
        this.props.authStore.isAuthenticated ? (
          <Children {...this.props} />
        ) : (
          <Redirect
            to={% raw %}{{
              pathname: "/login",
              state: { 
                  from: this.props.location 
              }
            }}{% endraw %}
          />
        )
      }
    />)
  }
}

export default PrivateRoute

最后剩下的就是修改 <Login />组件，如果用户未登录，就显示登录按钮，否则自动跳转回去之前的页面(真实情况是，已经登录后，就不能再重新登录了)

class Login extends React.Component {
  state = {
    redirectToReferrer: false
  };

  login = () => {
    fakeAuth.authenticate(() => {
      this.setState({ redirectToReferrer: true });
    });
  };

  render() {
    const { from } = this.props.location.state || { from: { pathname: "/" } };
    const { redirectToReferrer } = this.state;

    if (redirectToReferrer) {
      return <Redirect to={from} />;
    }

    return (
      <div>
        <p>You must log in to view the page at {from.pathname}</p>
        <button onClick={this.login}>Log in</button>
      </div>
    );
  }
}

基于角色（role）的组件访问权限（使用 HOC）

这样后端只要获取到用户的 role 就可以了。

https://hackernoon.com/role-based-authorization-in-react-c70bb7641db4 好文，写作风格用词等很不错, 含金量很大

注意两个词区别, 看起来挺像的

Authentication 身份认证 一般指登录 who are you Authorization 授权 指是否有权限访问某个页面/资源 are you allowed to see that

我们还需要让组件 self contained, reusable

第一步 在路由里传入 roles 属性

<Route> 组件添加属性 allowed(值为可访问的 roles 数组)

//Router Configuration
<Route
  allowed={['manager', 'admin']}
  path="feature"
  component={YourRoute}
/>

然后<YourRoute>组件内有一个可复用的<Authorization>组件，继续传入从父组件得到的props 数据中的 allowed.以及当前 user

// Route handler
class YourRoute extends React.Component {
    render() {
        return <Authorization allowed={this.props.allowed} user={this.state.user}>

第二步 roles 和组件关联

在每条路由里面传入角色列表给组件去处理并不是很好。我们想把这个角色列表下放到组件里。

使用 **HOC (高阶组件)**Authorization，传入待封装的组件WrappedComponent和参数allowedRoles，处理权限，然后返回被封装过的组件。（如果权限不符合，统一返回 No page for you）

// Authorization HOC
const Authorization = (WrappedComponent, allowedRoles) =>
  return class WithAuthorization extends React.Component {
    constructor(props) {
      super(props)

      // In this case the user is hardcoded, but it could be loaded from anywhere.
      // Redux, MobX, RxJS, Backbone...
      this.state = {
        user: {
          name: 'vcarl',
          role: 'admin'
        }
      }
    }
    render() {
      const { role } = this.state.user
      if (allowedRoles.includes(role)) {
        return <WrappedComponent {...this.props} />
      } else {
        return <h1>No page for you!</h1>
      }
    }
  }

现在<YourRoute>就可以用这个 HOC 来封装它了

// Route handler
class YourRoute extends React.Component {
  render() {
    return <div>
      /* the rest of your page */
    </div>
  }
}

export default Authorization(YourRoute, ['manager', 'admin'])

建立“角色与权限组”，简化管理

目前虽然可以用Authorization()传入组件、roles列表，来给一个组件绑定访问权限。但如果有多个路由，有不同的权限要求呢？

  • /users 允许 admin manager user 三种角色访问
  • /users/:id/edit 允许 manager 和 admin 两种角色访问
  • /users/create 只允许 admin 这一种角色访问

就会写很多类似的代码

Authorization(Users, ['admin', 'manager', 'user'])
Authorization(EditUser, ['admin', 'manager'])
Authorization(CreateUser, ['admin'])

不仅写起来繁琐，每次都要输入几个角色的字符串，万一以后想把 user 改名成 member，要大量改动代码。 而且不直观，全是Authorization，权限的级别要去看参数才知道。

现在把 Authorization 这个 HOC 的两个同等级参数（组件，roles 数组）中，第二个参数提升一下，变成一个函数，并且返回一个 HOC

const Authorization = (allowedRoles) =>
  (WrappedComponent) =>
    return class WithAuthorization extends React.Component {

通过改造，可以通过传入不同的 roles 参数，返回不同的 HOC ，这样生成我们需要的“角色与权限组”,非常方便集中管理、扩展组合不同的权限组

比如下面的三个：

const User = Authorization(['user', 'manager', 'admin'])
const Manager = Authorization(['manager', 'admin'])
const Admin = Authorization(['admin'])

然后路由里就可以不需要传入 roles, 而直接用这几个 HOC (角色与权限组)，类似component={Manager(EditUser)}

<Route path="/" component={App}>
    <Route path="users" component={User(Users)}>
      <Route path=":id/edit" component={Manager(EditUser)} />
      <Route path="create" component={Admin(CreateUser)} />
    </Route>
  </Route>

相当于

<Route path="/" component={App}>
    <Route path="users" component={Authorization(['user', 'manager', 'admin'])(Users)}>
      <Route path=":id/edit" component={Authorization(['manager', 'admin'])(EditUser)} />
      <Route path="create" component={Authorization(['admin'])(CreateUser)} />
    </Route>
  </Route>

如果 react 项目在某个子目录下

把项目放到 http://www.sample.org/app/ 而不是默认的根目录， 会发现把文件部署过去后，点击所有链接，会自动跳到根目录

例如本来希望链接是 http://www.sample.org/app/something

结果链接是 http://www.sample.org/something ,丢失了 app/

诡异的是，页面访问正常（history router 的功劳。。。），因为实际上没有发生服务器/后台的路由跳转

要想让链接自动带这个子目录，只要在使用 Router 组件的时候添加 basename参数即可

<Router basename={'/app'}>

https://medium.com/@svinkle/how-to-deploy-a-react-app-to-a-subdirectory-f694d46427c1

后端配合，当页面停留在子路由时，刷新不会提示 404

默认情况下，我们必须 先 进入 SPA（React & Vue）等的“入口文件”，也就是根路由，例如 /

然后通过点击动作，在子路由之间跳转，如 /test1 /test2

如果直接在浏览器里输入 /test1,或者前端路由跳转到 /test1时刷新浏览器， 会提示 404。 这是因为刷新动作会请求后端路由，而后端路由并没有匹配的规则。

解决办法：

(1)如果是 Laravel 等自带路由系统的框架，可以写一个路由规则（略，参看..） (2)简单地把应用所有文件放到某个目录，然后在 Nginx 里写一个 rewrite，让所有子路由请求全部跳回首页，让前端路由去接管处理

location / {
  try_files $uri $uri/ /index.html;
}

如果是一个子目录

location /reactApp {
    root /var/www;  // 注意，是这个子目录的上级主目录
    #index  index.html;
    try_files $uri $uri/ /reactApp/index.html;
}

这时候打开页面，会提示 css 文件 404.

分析 build/index.html, 里面的路径是“当前路径”（相对） ./sub/path/static/js/xxx.css 当然找不到了

需要修改 build 出来的index.html里 css 和 js 的路径为绝对路径/reactApp/.

打开 package.json 的 homepage, 把默认的 ./改成 /reactApp 重新 build，现在 /build/index.html里的路径是绝对路径了/reactApp/static/css/xxxx.css

如果需要动态改变，例如 build 到不同的子目录，可以使用

https://github.com/facebook/create-react-app/pull/937

PUBLIC_URL=https://www.your.domain.com npm run build

或修改 config/webpack.config.dev.js的const publicPath = '/'; (未测试)
