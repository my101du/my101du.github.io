title: 一台电脑两个浏览器分别访问同一个域名的本地绑定hosts和公网IP permalink: two-browser-visit-a-domain-localhost-and-website tags:

  • 技能

标题有些拗口，但做 web 开发的同学应该更容易理解，因为很可能你会经常遇到这个问题。

假设一个场景：需要对某个文件进行一些修改，然后分别在“本机环境”和“远程环境”上对比查看，观察这同一个修改在不同环境下显示效果是否完全一致（例如 PHP版本不一致、内存的大小等都有可能影响运行结果）。

为便于理解，以下测试域名均使用 www.test.com，本地环境 ip 用 127.0.0.1，远程环境 ip 用 111.222.333.444

按照之前的流程，我们需要进行下面几个步骤：

  1. 本地修改完代码文件
  2. 编辑 /etc/vhosts 文件，绑定 127.0.0.1 www.test.com
  3. 打开浏览器，查看代码在“本机环境”的效果
  4. 再次修改 vhosts 文件，绑定 111.222.333.444 www.test.com
  5. 把修改后的代码文件上传到 远程服务器
  6. 打开浏览器，查看代码在“远程服务器”的效果
  7. 对比完成后，如果有问题，可能又要重新进行一次上述步骤

非常繁琐有没有！

那么我们下面要做的，就是让两个浏览器，在访问相同域名的时候，一个走“本地服务器”，一个走“远程服务器”，并且不需要进行 vhosts 文件的修改！

浏览器优先使用“代理”还是“vhosts”？

1. 一个浏览器在访问某个域名的时候，如果遇到两个对 DNS 解析过程的“干扰”，先执行谁？

  • 代理服务器说：因为你走的是代理，而我是使用的公共 DNS 解析服务，因此 www.test.com 对应的 ip 地址是公网的111.222.333.444。
  • vhosts 文件说：因为我定义了一个规则127.0.0.1 www.test.com，因此www.test.com对应的 ip 地址是本地 127.0.0.1

2. 搭建本地服务器环境

我们做个测试，首先在本地服务器环境下建立一个测试站点www.test.com，这里以 Nginx 的虚拟主机配置文件为例

server
{
	listen       80;
	server_name www.test.com;
	index index.html index.htm;
	root  /home/wwwroot/www_test_com;
}

/home/wwwroot/www_test_com/index.html 文件是很简单的一个字符串

<h1>I'm a local site</h1>

3. 然后定义一条 vhosts 绑定规则

编辑 vhosts 文件，添加下面一条绑定规则

127.0.0.1 www.test.com

ping一下，发现已经指向本地了。

~ ping www.test.com PING www.test.com (127.0.0.1): 56 data bytes 64 bytes from 127.0.0.1: icmp_seq=0 ttl=64 time=0.087 ms 64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.079 ms 64 bytes from 127.0.0.1: icmp_seq=2 ttl=64 time=0.101 ms 64 bytes from 127.0.0.1: icmp_seq=3 ttl=64 time=0.091 ms

然后打开浏览器访问 www.test.com域名，发现已经正常指向到本地服务器了

4. 然后上代理！

打开 shadowsocks 这个代理客户端，使用“自动代理模式”，然后选择“编辑自动模式的PAC”，在gfwlist.js文件顶部添加一行，让 www.test.com 域名走代理。

var domains = {
  "test.com":1,
  "google.com":1,
略

重启浏览器，发现已经使用公网的DNS，提示一个访问错误（即使这时候 ping 这个域名，仍然显示为 127.0.0.1）

5. 结论

也就是说，“代理”的优先级是高于“vhosts 文件”的。

让 Chrome 浏览器绕过代理优先级，访问本地服务器

因此对于某个测试域名，如果在代理里设置了某个域名的规则，会导致全部的浏览器都会走这条规则。

比方说 www.test.com 在代理里，在 chrome safari 里都会显示远程服务器的效果

然而我们希望能在两个浏览器 chrome/safari 之间，同时对比 本地开发机器/远程服务器 的同一个域名

  1. 在 chrome 里查看 www.test.com 的本地效果
  2. 在 safari 里查看 www.test.com 远程效果

1. 首先需要安装一个 Chrome 的插件

Proxy SwitchyOmega

2. 然后打开这个插件的选项界面，新建一个“代理服务器”的情景模式

点击“新建情景模式”，建立一个**“代理服务器”**，取名叫“本地测试域名绕过shadowsocks代理”（为了理解方便，所以取这么长 O(∩_∩)O哈哈~）。

然后按下图填写信息，代理协议选 SOCKS5，代理服务器选127.0.0.1，代理端口选1080（实际上就 是我们 shadowsocks 的默认代理参数）

关键（测试域名绕过代理）

然后在 不代理的地址列表 里填写我们想走本地的测试域名通配符 *.test.com (二级域名是一个星号)

记得一定要点击左侧的“应用选项”按钮

如果启用这个模式，那么所有访问都会走代理服务器 socks5 127.0.0.1:1080，就相当于“全局模式” (注意，这时我的 ss 里是用的自动模式，没有走全局哦！)

说明 SwitchyOmega 是跑了代理，但是没有一个“过滤列表”来区分哪些走代理，哪些不需要。 这就导致如果我们的代理服务器在国外，在访问百度等国内网站，也会变慢（从国外绕回来）。

那就来为 SwitchyOmega 加一个过滤列表吧（就对应 ss 的自动模式里的 PAC gfwlist）

3. 然后再建立一个“PAC情景模式”，作为等下自动切换的“默认模式”

点击“新建情景模式”，选择PAC情景模式（还记得我们刚刚建立了一个代理服务器模式么？）名称“和shadowsocks翻墙域名列表保持一致”

把 shadowsocks 的“编辑自动模式的PAC”里那个 gfwlist.js 里的内容全部复制进来(或者用在线地址下载和更新)，点击“应用选项”

实际上就是实现了 ss 客户端的“自动代理模式”+PAC 列表

备注：

如果能在 PAC 模式添加一个“绕过地址”的参数就好了，就不用上面第一步和下面的第三步了。 可惜没有，只能这么复杂。。。

4. 自动识别不同域名，默认不走代理直连，在 PAC 列表里的

点击“新建情景模式”，选择**“自动切换模式”**，命名为“自动识别域名”。

如下图设置，点击+添加条件按钮，把测试域名添加进去，情境模式选择“本地测试域名绕过shadowsocks代理”。

“默认情境模式”一定要选“和shadowsocks翻墙域名列表保持一致”（PAC），否则我们的 chrome 就不能翻墙了…… direct 就是不走 ss 代理

最后点击“应用选项”

5. 完成

这样我们就可以让 chrome 访问 www.test.com 这个域名是走的“本地服务器”，而用其他的浏览器（例如 safari、firefox 等）访问这个域名，则使用代理的方式访问公网服务器。

对比一下看看！
