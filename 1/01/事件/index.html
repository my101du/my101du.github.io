事件

Laravel 事件机制实现了一个简单的观察者模式，让我们可以订阅和监听应用中出现的各种事件.

事件类(Event)通常放在app/Events目录下，而对应的侦听类（Listener）保存在app/Listeners目录下

事件机制是一种很好的应用解耦方式。 一个事件可以拥有多个互不依赖的监听器。例如每次把用户的订单发完货后都希望给他发个 Slack 通知。 可以发起一个 OrderShipped 事件，它会被监听器接收到再传递给 Slack 通知模块，这样你就不用把订单处理的代码跟 Slack 通知的代码耦合在一起了

注册事件/监听器

app/Providers/EventServiceProvider（注册事件和监听器对应关系） app/Events/（事件） app/Listeners(监听器)

手动创建

可以在 EventServiceProvider (app/Provider/EventServiceProvider)中注册所有的事件侦听器。它的 listen 属性是一个数组，包含所有的事件（键）以及事件对应的监听器（值）

例如下面的代码，表示当发生PodcastWasPurchased事件时，App\Listeners\EmailPurchaseConfirmation这个监听器会被触发并作出相应处理。

protected $listen = [
    'App\Events\PodcastWasPurchased' => [
        'App\Listeners\EmailPurchaseConfirmation',
    ],
];

除了上面在$listen数组中注册，还可以在boot方法中注册闭包事件

//注册应用程序中的任何其他事件
public function boot()
{
    parent::boot();

    Event::listen('event.name', function ($foo, $bar) {
        //
    });
}

通配符事件监听器

在注册监听器时使用 * 通配符参数，它让你在一个监听器中可以监听到多个事件。通配符监听器接受的第一个参数是事件名称，第二个参数是整个的事件数据

Event::listen('event.*', function ($eventName, array $data) {
    //
});

生成 Event/Listener

运行下面的命令会生成所有列在EventServiceProvider中列出的所有事件和监听器（如果存在就不会去修改它）,这样就不要我们一个个文件去手动创建了。

php artisan event:generate

定义事件

事件类就是一个包含与事件相关信息数据的容器.本例生成的 OrderShipped 事件接受一个 Eloquent ORM 对象。

如下看到，它没有包含其它逻辑。它仅只是一个被构建的 Order 对象的容器

注意use Illuminate\Queue\SerializesModel，使用了 SerializesModels trait 的事件将会优雅的序列化任何的 Eloquent 模型

namespace App\Events;

use App\Order;
use Illuminate\Queue\SerializesModels;

class OrderShipped
{
    use SerializesModels;

    public $order;

    /**
     * 创建一个事件实例。
     *
     * @param  Order  $order
     * @return void
     */
    public function __construct(Order $order)
    {
        $this->order = $order;
    }
}

定义监听器

事件监听器在 handle 方法中接受了事件实例作为参数，前面运行 event:generate 命令将会在事件的 handle 方法中自动加载正确的事件类和类型提示。在 handle 方法中，你可以运行任何需要响应该事件的业务逻辑

你的事件监听器也可以在构造函数(__construct)中对任何依赖使用类型提示。所有的事件监听器会经由 Laravel 的 服务容器 做解析，所以所有的依赖都将会被自动注入：

namespace App\Listeners;

use App\Events\OrderShipped;

class SendShipmentNotification
{
    /**
     * 创建事件监听器。
     *
     * @return void
     */
    public function __construct()
    {
        //
    }

    /**
     * 处理事件
     *
     * @param  OrderShipped  $event
     * @return void
     */
    public function handle(OrderShipped $event)
    {
        // 使用 $event->order 来访问 order ...
    }
}

停止事件传播

如果希望停止一个事件传播到其他的监听器（事件会依次在多个监听器之间传递下去）。这时你可以通过在监听器的 handle 方法中返回 false 来实现。

队列化事件监听器

如果在事件监听器中要进行一些耗时任务，可以把它们放入“队列”中（在使用队列化监听器之前，需要在服务器或者本地环境中配置 队列 并开启一个队列监听器）

只需增加 ShouldQueue 接口到你的监听器类，就可以对监听器进行队列化。（由 Artisan 命令 event:generate 生成的监听器已经将此接口导入到命名空间了，因此你可以直接使用它）

当然，可以在监听器类中定义 $connection 和 $queue 属性来自定义队列的连接和名称

namespace App\Listeners;

use App\Events\OrderShipped;
use Illuminate\Contracts\Queue\ShouldQueue;

//增加 `ShouldQueue` 接口到你的监听器类 implements ShouldQueue
class SendShipmentNotification implements ShouldQueue
{
    /**
     * 队列化任务使用的连接名称。
    */
    public $connection = 'sqs';

    /**
     * 队列化任务使用的队列名称。
     */
    public $queue = 'listeners';

    //
}

当事件被监听器调用时， 事件分发器会使用 Laravel 的 队列系统 自动将它进行队列化.如果监听器通过队列运行且没有抛出任何异常，则已执行完的任务将会自动从队列中删除

手动访问队列

需要手动访问底层队列任务的 delete 和 release 方法，可以使用 Illuminate\Queue\InteractsWithQueue trait 来实现

class SendShipmentNotification implements ShouldQueue
{
    use InteractsWithQueue;
    
    public function handle(OrderShipped $event)
    {
        if (true) {
            $this->release(30);
        }
    }

处理失败任务

队列化的事件监听器可能失败（队列监听器任务执行次数超过在工作队列中定义的最大尝试次数，监听器的 failed 方法将会被自动调用，它接受事件实例和失败的异常作为参数）

public function handle(OrderShipped $event){}

public function failed(OrderShipped $event, $exception)
{
    //
}

触发事件

通过传递一个事件实例给 event() 辅助函数.将会把事件分发到它所有已经注册的监听器上。因为 event 函数是全局可访问的，所以你可以在应用中的任何地方调用它

class OrderController extends Controller{
    public function ship($orderId){
        $order = Order::findOrFail($orderId);
        
        // 订单的发货逻辑...
        
        event(new OrderShipped($order));
    }
}

测试时，不用真的触发监听器就能断言事件类型是很有用的。 Laravel 内置的测试辅助方法 能让这件事变得很容易。

我们只需要触发事件，并在event()里传入具体的内容，因为对应的监听器会自动响应事件并作出处理。

事件订阅者

虽然可以在 EventServiceProvider 的 $listen 中注册所有的 事件/监听器。但是当条目数量增加的时候，更好的办法是将它们进行“分组”管理和集中处理。

例如与用户有关的所有事件（登录、注销）、与订单有关的所有事件（发货、物流信息变化）等。

编写事件订阅者

事件订阅者：

  • 在自身内部订阅多个事件
  • 在单个类中定义多个事件处理器，
  • 进行事件分发。

如下例，定义一个UserEventSubscriber的事件订阅者，然后定义一个 subscribe 方法（它接受一个事件分发器的实例 $events 作为参数），在 subscribe 方法内部多次调用事件分发器$events的listen()方法来注册多个事件监听器.

listen()方法第一个参数是事件类，第二个参数是在当前订阅者类中实现的某个事件监听方法（onUserLogin， onUserLogout）。把之前分散在多个 Listener类 中的 handle方法，统一集中放在这个事件订阅者内部来处理。

事件分发器类 Illuminate\Events\Dispatcher

namespace App\Listeners;

class UserEventSubscriber
{
    
    public function onUserLogin($event) {}

    public function onUserLogout($event) {}

    /**
     * 为订阅者注册监听器。
     *
     * @param  Illuminate\Events\Dispatcher  $events
     */
    public function subscribe($events)
    {
        $events->listen(
            'Illuminate\Auth\Events\Login',
            'App\Listeners\UserEventSubscriber@onUserLogin'
        );

        $events->listen(
            'Illuminate\Auth\Events\Logout',
            'App\Listeners\UserEventSubscriber@onUserLogout'
        );
    }

}

注册事件订阅者

一旦订阅者被定义，它就可以被注册到事件分发器中

因为现在把多个事件和监听器的注册都放到了事件订阅者中，所以现在不用再在 EventServicePriver的$listen里面去一个一个订阅事件和监听器了。

直接在$subscribe属性里订阅一个事件订阅者来批量实现。

class EventServiceProvider extends ServiceProvider
{
    /**
     * 应用中事件监听器的映射。
     */
    protected $listen = [
        //
    ];

    /**
     * 需要注册的订阅者类。
     */
    protected $subscribe = [
        'App\Listeners\UserEventSubscriber',
    ];

protected $listen = [
