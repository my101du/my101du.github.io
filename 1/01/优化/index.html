性能测试

注意

我一直测的是 https, 所以前后变化非常小，还在检查部署 ssl 后对性能的真实影响还是 ab 的不支持导致（因为把网站改回 http 后，带查数据库的测试页面，参数 -n 1000 -c 100 ，能跑到 58 ,不缓存配置和路由在 46）

测试工具把 ab 换成 go get -u github.com/rakyll/hey 后再试，终于能排除 https 的影响了

http://laravelacademy.org/post/2398.html

如果没有 opcache 等优化，rps 是很低的。。。

opcache 与 php7.x

mac 上 brew 安装的 php71 brew install php71, 用 phpinfo() 打印出来后，发现 Configure Command 里有参数 –disable-opcache

运行 opcache_get_status() 返回错误

添加 opcache brew install php71-opcache 提示包不存在

php71 opcache No similarly named formulae found

google 了很久，发现没有什么有价值的信息. 后来在这里发现 https://stackoverflow.com/questions/49579961/macos-php71-became-php7-1-with-brew。

安装 php 的时候，不要用 php71 这个名字，要用 php@7.1, 好像是包被移动到不同的仓库了。重新安装吧

brew services stop php71
brew remove php71

brew install php@7.1
brew services start php71

重新打印phpinfo(),成功加载 opcache 扩展

php -v 
# 还是旧版本，what?

还要 link 一下

brew link php@7.1 --force
Linking /usr/local/Cellar/php@7.1/7.1.17... 25 symlinks created

If you need to have this software first in your PATH instead consider running:
  echo 'export PATH="/usr/local/opt/php@7.1/bin:$PATH"' >> ~/.zshrc
  echo 'export PATH="/usr/local/opt/php@7.1/sbin:$PATH"' >> ~/.zshrc

无法删除系统自带的php，要关闭 mac 的 SIP 特性（/usr/bin这个目录被rootless保护了）

which php
/usr/bin/php

rm /usr/bin/php
#rm: /usr/bin/php: Operation not permitted

重启mac，按住 Cmd+R，直到出现苹果logo。Utilities(实用工具) > Terminal(终端).输入命令 csrutil disable

再次运行 ab 性能测试，rps 略有提高。。。 完全没达到想要的效果啊！摔！

数据库处理

如果一次批量 insert 太多元素（一个巨大的数组），可能出现两种情况导致失败：

（1）运行超时 Maximum execution time of 30 seconds exceeded

插入的元素中，可能有很长的字符串，并且元素很多

使用如下的函数，把运行运行的时间改成不限制。

set_time_limit(0);

但是不建议这样，因为又可能会触发 nginx 的 504 错误。

应该想办法从程序上来优化

使用 队列 来处理

（2）提示 prepared statement contains too many placeholders.

试过把每一个数组的元素都改成空白字符串，

查资料得知和 MySQL Prepared statement 最多支持65535个占位变量，将要插入的数组拆分处理即可。。。

https://www.jianshu.com/p/33bda6dc427b http://raising.iteye.com/blog/2296092

这个数字怎么算？

可是我发现我入库的表也才40+个字段，应该不至于吧，但是……

在批量入库时，Mysql会将insert语句拼装成如下格式： Sql代码 收藏代码 insert into tablename(字段1, 字段2, 字段3, ……, 字段45) values(:字段1, :字段2, :字段3, ……, :字段45), (:字段1, :字段2, :字段3, ……, :字段45), (:字段1, :字段2, :字段3, ……, :字段45), ……, (:字段1, :字段2, :字段3, ……, :字段45)
比如我批量入库的对象是3000+条记录，虽然我把它封装成一个List集合，但是集合里面每个元素都有45个key（一个key对应一个字段）。sql被拼装成上述格式后，占位符的个数达到3000*45=135000，大于mysql对占位符的最高限制65535。 所以需要注意的是，批量入库时，占位符的个数不再是45，而是45与条数的乘积。
