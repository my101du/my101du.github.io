使用Linux+Nginx(Apache)+MySQL+PHP搭建网站

slug: linux-nginx-mysql-php-apache-set-up date: 2016/05/25 10:10:10 tags:

  • 运维

目前，有不少网站站长都不再满足于租用一个和别人共用的虚拟主机（指开箱即用，或者简单配置后即可运行的网站），因为往往这些虚拟主机在开始的时候能满足需求，随着访问人数攀升，以及个性化定制需求的增加，各种不灵活的掣肘就浮现出来了。拥有一台自己拥有完全控制权的独立服务器（或 VPS，或现在流行的云主机），用来搭建一个网站以及运行有意思的服务（爬虫、远程备份，等等），就成为进阶站长和技术爱好者的目标。

又或者你只是想先在自己的电脑上搭建一个 PHP 的运行环境进行学习、开发。

都可以在这里找到细致而详尽的答案。

本文包括在三种主流的操作系统（Linux & Windows & OS X）上搭建 Nginx+PHP+MySQL 环境的过程。因为 LEMP（Linux + Nginx + MySQL + PHP）实际上已经是现在很多中小型网站运行的主流环境了，所以我们重点笔墨放在 Linux 上，我们也建议 Windows 的开发者尽可能使用虚拟机来安装 Linux，可以避免遇到很多莫名其妙的问题。

本文并未涉及服务器的选购、 Linux 安装配置、以及 Nginx+PHP+MySQL 的性能优化等知识。 如果需要了解这些内容 ,请查看我们相关的教程：

  • 如何选购合适的独立服务器-vps-云主机
  • Linux安装配置与常用命令
  • 基于Linux+Nginx(Apache)+MySQL+PHP的网站优化

最简单的方式：一键安装

XAMPP

跨平台的 Apache + MySQL(MariaDB)+ PHP，并自带了 Ftp 服务端、PhpMyAdmin等。

下载 Linux 版本（.run 后缀名），安装后会在 /opt/lampp

# 一些控制命令（修改了Apache、PHP、MySQL的配置文件后要重启）
./ctlscript.sh help
       ./ctlscript.sh (start|stop|restart|status)
       ./ctlscript.sh (start|stop|restart|status) mysql
       ./ctlscript.sh (start|stop|restart|status) proftpd
       ./ctlscript.sh (start|stop|restart|status) apache

Linux 下的 LNMP 环境一键安装包

如果你没有太多时间去研究各种软件包版本和库之间的依赖关系，而且对性能调优不太熟悉，只想尽快搭建一个可以快速运行起来的网站的话建议选择一键包。 一键包除了可以简化繁琐和容易出错的安装过程外，还有很强大的参数调整方式（通过命令行或图形界面）。 但也有缺陷，那就是安装的时候可定制型不强，一些特殊的模块可能并没有默认安装。

我最常用的是军哥的一键包

这个一键包的作者有多年的运维经验，非常建议多读读安装脚本的代码，可以从中学到很多的知识。 在 http://lnmp.org 下载，解压并运行安装脚本即可，安装过程中会有详细的提示，用于选择 PHP 版本、MySQL版本、默认 root 密码等等。

$ wget -c http://soft.vpser.net/lnmp/lnmp1.3-full.tar.gz && tar zxf lnmp1.3-full.tar.gz && cd lnmp1.3-full && ./install.sh lnmp

最新版的一键包支持 PHP7(对比PHP5有几倍的性能提升，但有部分函数不兼容一些比较古老的PHP框架/CMS，在稳定的生产环境中还是建议使用 PHP5)。

除了安装基本的 LEMP 环境外，还可以添加 ftp 服务器、memcached 内存缓存服务等功能。

由于这个一键包没有提供图形界面的管理面板，所以只能通过命令行去启动服务、添加主机等操作。

LNMP无法删除或更改权限，显示：

rm: cannot remove `.user.ini’: Operation not permitted

无法删除“.user.ini”文件解决方法,运行后删除即可

chattr -i /home/wwwroot/default/.user.ini

如果是需要修改文件，记得修改完以后运行：

chattr +i /home/wwwroot/default/.user.ini

如果你对命令行并不熟练，也可以选择WDCP、AMH等带有图形界面的管理面板，其中 好像WDCP在阿里云上用得比较多；AMH也是一个非常不错的环境，特点是有一个优化过的 MySQL 管理器 。

默认不会安装 ext_ldap ext_filenfo 等包

在laravel应用 composer install 会报错。需要手动安装

以 ldap 为例

# 先进入 src 目录, 解压 php 压缩包（bz2格式）
cd ~/lnmp1.5/src
tar -jxvf xx.tar.bz2

# 进入需要安装的扩展目录
cd php-7.1.18/ext/ldap

# 然后依次运行 （--with-ldap 可以不带？ fileinfo 测试）
/usr/local/php/bin/phpize
./configure --with-php-config=/usr/local/php/bin/php-config --with-ldap

如果报错

configure: error: Cannot find ldap.h

则安装下面的包后再运行 configure

yum install openldap openldap-devel  -y

如果继续提示

configure: error: Cannot find ldap libraries in /usr/lib

则复制一些文件后重新 configure

cp -frp /usr/lib64/libldap* /usr/lib/

运行make 命令

make && make install

提示如下，记住这个路径

Installing shared extensions: /usr/local/php/lib/php/extensions/no-debug-non-zts-20160303/

打开/usr/local/php/etc/php.ini 添加fileinfo.so扩展

extension = "ldap.so"

重启确保生效,

lnmp retstart

再运行下面的命令查看模块列表

php -m

Windows 下的一键安装包

XAMPP，比较大，只有 Apache

UPUPW (有 Apache 或 Nginx 版本)，可以自己搭配 PHP & MySQL 版本

Apache 使用别的目录做虚拟主机目录时，403 Forbidden问题

#定义个目录
Define VHOSTS_ROOT "/mnt/share"

# 编辑 conf/httpd-vhosts.conf
# 使用了其他目录作为虚拟主机目录，这段是最重要的
<Directory "${VHOSTS_ROOT}">
    Options Indexes FollowSymLinks
    AllowOverride None
    Require all granted
</Directory>

CentOS 系统安装 Nginx + PHP & MySQL

对一些有软件洁癖的技术爱好者，或者一些老鸟来说，用一键包往往难以满足他们从头到尾控制安装过程的满足感。下面就分别介绍 CentOS 和 Ubuntu 这两个最流行的发行版上安装 LEMP 环境的方法。

这里默认的版本为 CentOS 7，部分在 CentOS 6 下有差异的地方附有说明。

添加 epel 源

#因为 nginx 没有在官方源里面，需要添加epel源
$ sudo yum install epel-release

#还需要安装下面的 rpm 包以便能安装 PHP5.5

#如果是 CentOS 7 
$ rpm -Uvh https://mirror.webtatic.com/yum/el7/epel-release.rpm
$ rpm -Uvh https://mirror.webtatic.com/yum/el7/webtatic-release.rpm

#如果是 CentOS 6
$ rpm -Uvh https://mirror.webtatic.com/yum/el6/latest.rpm

安装必需的软件和库

#下面的命令安装包含了 `pcre` `openssl` 等 `nginx` / `tengine` 必须要用到的库
$ yum -y install ntp make openssl openssl-devel pcre pcre-devel libpng libpng-devel libjpeg-6b libjpeg-devel-6b freetype freetype-devel gd gd-devel zlib zlib-devel gcc gcc-c++ libXpm libXpm-devel ncurses ncurses-devel libmcrypt libmcrypt-devel libxml2 libxml2-devel imake autoconf automake screen sysstat compat-libstdc++-33 curl curl-devel

安装Nginx

用 yum来安装 Nginx

$ sudo yum install nginx

按照提示输入 yes 后开始安装。安装完毕后，Nginx 的配置文件在 /etc/nginx 目录下。其中 sites-enabled目录中保存激活的虚拟主机配置文件。

#使用以下命令启动 Nginx：
$ sudo systemctl start nginx

有时候可能存在问题：启动时或运行配置参数重新加载的命令 nginx -s reload 时报错

nginx: [error] open() “/run/nginx.pid” and failed

需要 -c 参数手动指定 nginx 的配置文件路径 /etc/nginx/nginx.conf

$ /usr/sbin/nginx -c /etc/nginx/nginx.conf

最后将Nginx设置为开机启动

$ sudo systemctl enable nginx.service

最后测试一下 nginx 服务是否正常运行。启动 nginx 服务后，通过浏览器访问http://localhost（本地开发环境）或 http://<外网IP地址> 来确定是否已经启动。如果一切正常，页面会显示一行

Welcome to Nginx

补充

更建议用 Tengine 来替代 Nginx，并用编译源代码的方式来安装，见后面的章节**“用 Tengine 替代 Nginx”**

开放防火墙中的端口

检查系统中 firewalld 防火墙服务是否开启

$ sudo systemctl status firewalld

如果已开启（显示 active (running)），我们需要修改防火墙配置，开启外网访问 80 端口。 修改 /etc/firewalld/zones/public.xml 文件，在 <zone></zone>一节中增加 <service name="nginx"/> 关于防火墙的设置，可以学习我们讲解 CentOS 的安装和配置有关教程。

保存后重新加载firewalld服务

$ sudo systemctl reload firewalld

安装 MariaDB（MySQL）

MariaDB是MySQL的一个开源分支，在 MySQL 被甲骨文公司收购后，由开源社区进行开发和维护，在CentOS 7的软件仓库中，已经默认将 MySQL更替为了MariaDB。

使用yum直接安装MariaDB

sudo yum install mariadb-server

#安装完成之后，执行以下命令启动MariaDB服务
sudo systemctl start mariadb

基本安全设置（root密码等）

MariaDB默认root密码为空，我们需要设置一下，执行脚本：

sudo /usr/bin/mysql_secure_installation

这个脚本会经过如下一系列的交互问答来进行MariaDB的安全设置。

首先提示输入当前的root密码： Enter current password for root (enter for none): 初始root密码为空，我们直接敲回车进行下一步。

Set root password? [Y/n] 设置root密码，默认选项为Yes，我们直接回车，提示输入密码，在这里设置您的MariaDB的root账户密码。

Remove anonymous users? [Y/n] 是否移除匿名用户，默认选项为Yes，建议按默认设置，回车继续。

Disallow root login remotely? [Y/n] 是否禁止root用户远程登录？如果您只在本机内访问MariaDB，建议按默认设置，回车继续。 如果您还有其他云主机需要使用root账号访问该数据库，则需要选择n。

Remove test database and access to it? [Y/n] 是否删除测试用的数据库和权限？ 建议按照默认设置，回车继续。

Reload privilege tables now? [Y/n] 是否重新加载权限表？因为我们上面更新了root的密码，这里需要重新加载，回车。

登录测试

完成后你会看到Success!的提示，MariaDB的安全设置已经完成。我们可以使用以下命令登录MariaDB：

mysql -uroot -p

#按提示输入root密码，就会进入MariaDB的交互界面，说明已经安装成功。
mysql>
mysql> show databases;
+--------------------+
| Database           |
+--------------------+
| information_schema |
| mysql              |
| performance_schema |
+--------------------+
3 rows in set (0.00 sec)

最后将MariaDB设置为开机启动

sudo systemctl enable mariadb

可能的问题：在连接外部数据库时提示无法连接

Error:Can’t connect to MySQL server on ‘10.211.55.2’ (13) Errno:2003

解决办法 关闭CentOS的 SELinux

setenforce 0
vi /etc/selinux/config
# 修改 SELINUX=enforcing 为 SELINUX=disabled

重装或替换 MariaDB 为 MySQL

卸载

#查看CentOS自带mysql是否已安装
yum list installed | grep mysql

#若有自带安装的mysql，卸载CentOS系统自带mysql数据库，若有多个依赖文件则依次卸载
yum -y remove mysql-libs.x86_64
yum -y remove [依次卸载其他文件]

#当结果显示为Complete！即卸载完毕

开始安装

#查看yum库上的mysql版本信息
yum list | grep mysql 或 yum -y list mysql*

#安装mysql-server，mysql(包括服务端和client)
yum -y install mysql-server mysql mysql-devel 

#查看刚安装mysql数据库版本信息
rpm -qi mysql-server

最后要重复之前安装过程中的 基本安全设置 和 登录测试 以及 配置和优化步骤。

安装PHP（默认是5.4，安装5.5的方法在后面一节）

#先安装PHP(默认5.4)

比较麻烦，建议直接安装 PHP5.5，自带有 Opcache

Ubuntu 系统安装 LEMP

Ubuntu 的软件源和 apt-get 命令非常强大，除了安装时自动解决或提示烦人的依赖关系，在需要卸载和和清理软件的时候也是很方便的。这可能和 CentOS 特别追求稳定，一般在生产环境配置好后就不怎么动了；而 Ubuntu 比较倾向于开发者日常使用，经常会更新软件和修改配置（当然 Ubuntu Server 也非常稳定）

以下均以 Ubuntu 14.04 LTS 为例。

安装必备的环境

sudo apt-get install build-essential
apt-get install libpcre3 libpcre3-dev
apt-get install libssl-dev

通过 apt-get 快速安装MySQL、Nginx

#先 update 一下
sudo apt-get update

#Nginx
sudo apt-get install nginx
sudo service nginx start
sudo service nginx restart

#MySQL 如果是 14.10 可能会提示“非稳定版……”，使用 mysql-server-5-5 即可
sudo apt-get install mysql-server 

# 如果上面出错，也可以下载repo再安装 MySQL
http://dev.mysql.com/downloads/repo/apt/
sudo dpkg -i /PATH/version-specific-package-name.deb
sudo apt-get install mysql-server

# 安装 MySQL 的系统数据库，以及初始化安全配置
sudo mysql_install_db
sudo /usr/bin/mysql_secure_installation

安装PHP

安装 PHP5

sudo apt-get install php5-fpm php5-mysql

# 编辑 php-fpm 配置，修改 listen（注释之前的 127.0.0.1:9000）,注意以后的 Nginx 虚拟主机都用这个
vi /etc/php5/fpm/pool.d/www.conf
listen = /var/run/php5-fpm.sock

# 重新启动 php5-fpm 服务
sudo service php5-fpm restart

安装 PHP7

$ sudo add-apt-repository ppa:ondrej/php

# 如果失败
$ sudo apt-get install -y language-pack-en-base
$ sudo LC_ALL=en_US.UTF-8 add-apt-repository ppa:ondrej/php

$ sudo apt-get update

为什么要使用最新版本的 MySQL？以及新旧版本数据库的环境兼容问题

以前我是一个“顽固派”，认为软件的版本够用就好，没必要追求最新版，所以在 Mac 上一直用着 5.1 版本。它体积小，不占太多硬盘空间，而且启动后台服务后占用内存也才 20M 不到，非常节省资源。

但是 MySQL 这个数据库不一样，最新版的会有更强大的性能（当然占用系统资源也会更多），以及更多的特性（虽然平时感觉不到），和对数据更好的兼容（毕竟现在数据比什么都重要，而且数据来源五花八门）。

但就是这个平时感觉不到的东西，最近坑了我一把。

我们的生产服务器上是跑的 5.5 版本，有一次把一个通过 hotcopy 方式生成的备份数据库还原到本地后，发现部分表无法打开。表现为在 GUI 工具里点击表名，弹出一个错误

Error : Incorrect information in file: ‘./my_database/my_table.frm’ error : Corrupt

以为是还原过程中出现了中断导致错误，于是进行修复

$ mysqlcheck -r my_database -uroot -p

还是一样，并且还发现这个库有更多的表都会报同样的错误。

以为是数据库引擎变化的问题（因为我本地 MySQL5.1 启用默认引擎为 MyISAM），远程到生产服务器上，查看数据表结构，也是默认 MyISAM 引擎，没有问题啊！

再打开之前还原过来的其他数据库（一个 discuz 论坛的数据库），却没有这个问题，所有表都正常。

仔细回忆下，唯一的不同就是：报错的这个数据库以及里面的一些表，是在生产服务器上的 MySQL5.5 环境下创建的，而那些不报错的，是很早就在 MySQL5.1 版本里创建，然后逐个版本升级、迁移上去的，“基础信息”没有变。

为了印证这个猜想，把这个在我电脑上报错的数据库压缩后发给同事，在他安装了 MySQL 5.5 的电脑上还原，结果一切顺利！

得，那就把本地的 MySQL5.1 升级吧！

替换上面命令中的数据库、数据表、用户名、主机名、密码，[database]使用 * 表示服务器上所有的数据库，[tablename]使用 * 表示数据库下所有的表。其中 ALL PRIVILEGES 是分配所有权限，有一定的安全隐患，一般情况下只需要分配需要的权限即可，以下是说明

1、数据部分 SELECT：是指允许读取数据 INSERT：是指允许插入和替换数据 UPDATE：是指允许更改数据 DELETE：是指允许删除数据 FILE：是指允许从数据中导入数据，以及允许将数据导出至文件

注意一个 Mac 下编译 openssl 的诡异问题

  4. 正确指定 openssl 路径参数后的编译报错 symbol(s) not found for architecture x86_64(中间有个停顿几秒的提示)，原因是nginx 在调用 openssl 的源码编译时, 调错了 configure 文件, 最终没能正确编译出需要的 openssl x86_64 库文件。 解决方法：在 nginx 编译时执行完 ./configure 命令后, 不要继续 make, 要先修改下 Makefile 文件

nginx: configuration file /usr/local/nginx/nginx.conf test is successful

# 重启
nginx -s reload

安装第三方模块

方法和上面差不多，只需要添加一个 --add-module 参数

./configure --prefix=/你的安装目录  --add-module=/第三方模块目录

# 例如
./configure  --with-pcre=/usr/local/src/lnmp/pcre-8.33 --with-zlib=/usr/local/src/lnmp/zlib-1.2.8 --with-openssl=/usr/local/src/lnmp/openssl-1.0.1e --with-http_gzip_static_module --with-http_ssl_module --add-module=../ngx_pagespeed-master
 
make
 
cp ./objs/nginx /usr/local/nginx/nginx
 
nginx -s reload

一次坑爹的经历——在OSX上的PHP环境上安装openssl扩展

slug: osx-and-php-install-openssl-extension date: 2015/12/18 10:10:10 categories:

  • server

事情是这样子的。 两年前，我最初在 OSX 10.9 上全部用手动编译的方式安装了 Nginx1.2.7 + PHP5.6.11，当时安装 PHP 时并未在 ./configure 命令中用 --with-openssl 把 openssl 扩展安装进去（因为工作中没有用到过）。 后来一直平滑升级到 10.10，10.11，由于安装的环境一直运行正常，所以也没有进行任何变动，只修改了一些配置参数。 因为最近需要学习 Laravel，在安装 Composer 的时候报错，提示一定要安装 OpenSSL 扩展才行，于是踏上了漫漫的打怪征程……

首先尝试 phpize 方式动态添加 openssl 扩展

一开始我理解错了，把 php 的 openssl 扩展和单独的 openssl 库两者搞混了。

首先是错误的方式——单独的 openssl 库

从 https://www.openssl.org/ 网页上下载了 openssl1.0.2d，解压后进行如下操作

$ cd Download/openssl-1.0.2d/
$ /usr/local/php/bin/phpize
# 这里会提示 Cannot find config.m4.按网上的说法复制一下就好了
$ mv config0.m4 config.m4
# 报错提示：文件不存在，喵了个咪的，这个目录下哪有什么 config0.m4 文件啊？其实是我搞错了

正确的方式 —— php 的 openssl 扩展

进入 PHP5.6 的安装源码包，进行操作（注意看扩展所在的目录是 /ext/openssl）

$ cd /php-5.6.11/ext/openssl
$ /usr/local/php/bin/phpize
$ mv config0.m4 config.m4      # 现在这一步是正常的，因为 config0.m4 文件存在
$ ./configure

然后就开始debug之路了，因为作为程序员，特别是还要自己搭环境的，人生不如意事常八九……

报错： ld: symbol(s) not found for architecture x86_64 错误

出现下面的错误

ld: symbol(s) not found for architecture x86_64

网上查到资料说有个 Xcode 的路径要做链接

ln -s /Applications/Xcode.app/Contents/Developer/Platforms/MacOSX.platform/Developer/SDKs/MacOSX10.11.sdk/usr/include /usr/include

呵呵，提示权限不够，加 sudo 也不行。原来是新版 MacOS 的 System Integrity Protection （SIP）功能禁止修改核心系统目录，我们要关闭它。

  1. 重启电脑
  2. 按住 CMD + R 键，进入安装模式
  3. 点击顶部菜单栏，进入“终端”
  4. 输入 csrutil disable
  5. 重启电脑 reboot

重新添加 xxxxxxx/include 目录的链接后，再次使用 phpize方法动态添加 openssl扩展，依然报错。

重新安装 PHP

既然上面的动态扩展方法不行，那就重新完整安装一遍 PHP 吧，在 configure阶段就把 openssl 扩展包含进去不就行了？

  1. 首先删掉所有的 /usr/local/php 目录
  2. 解压 PHP 源码包，在 configure 命令中添加进 --with-openssl 选项
  3. make 和 make install

这次报错如下

fatal error: ‘openssl/evp.h’ file not found

咦，怎么没找到？**使用 brew 重新安装 openssl **看看（当然也可以编译前面下载的）

$brew install openssl
$brew link openssl --force

然后重新进入php的安装目录，在 configure命令中添加 --with-openssl=/usr/local，指定 openssl的 路径

还是报错 evp.h' file not found，接着 google，发现有人说要重新安装 libxml2

$ brew install libxml2
$ brew link libxml2 --force

再次进入 php 目录 make clean，make，然而报错依旧……

最后办法，大道至简，祭出 brew（同时升级 Xcode）

算了，既然不让手动编译，那就用 Mac 下最牛逼的 brew 来安装 PHP 吧。

$ brew install php56

报错

Error: You must `brew link libpng freetype jpeg unixodbc’ before homebrew/php/php56 can be installed

To force the link and overwrite all conflicting files:

大概是我之前手动编译安装的 libpng、freetype 什么的都不行了，要重新来一发。依次安装提示里的几个库（下面只以 freetype 为例）

$ brew install freetype
$ brew link --overwrite freetype

还是不行！！！

在 google 上找来找去，基本上翻遍了能看懂的所有解决方法，最后有人说，这是因为OSX 升级后，brew 没有 update，以及 Xcode 版本太老的问题……

首先进入 App Store，找到 Xcode，点击“安装”，3个小时过去了，终于安装完毕，再看版本号是 7.2，真是奇怪，我之前一直是 7.0 版本，居然都从来不提示我升级？？OSX 的软件质量真是堪忧啊。升级完毕 Xcode，继续进入终端：

$ brew update
$ brew install php56

提示如下

configure: error: in /private/tmp/icu4c20151219-7556-wtja8m/icu/source': configure: error: C compiler cannot create executables See config.log’ for more details

Agreeing to the Xcode/iOS license requires admin privileges, please re-run as root via sudo.

意思大概是还需要启动 Xcode ，并在一开始弹出的同意协议窗口里点确定……好吧，打开 Xcode，点了一下同意协议，关闭 Xcode。

再次 brew install php56

终于成功了!!!

以下是一些比较有用的信息，和自己编译的 PHP 在文件路径有一点点不一样，PHP安装在 /usr/local/Cellar/php56/5.6.16/这个目录里面。

The php.ini file can be found in: /usr/local/etc/php/5.6/php.ini

PATH=”/usr/local/sbin:$PATH”

The control script is located at /usr/local/opt/php56/sbin/php56-fpm

并且已经自动在 /usr/local/bin 里为 php、phpize 等命令创建了链接，brew 大法好！运行一下看看成功没有

$ php -m
# 如果看到列出的扩展里有 openssl 就可以了

fasgcgi两种链接方式

nginx通过fastcgi链接php-fpm有两种方式：

tcp 和 unix domain socket

nginx server模块设置fastcgi链接方式：

tcp方式：

fastcgi_pass 127.0.0.1:9000;

socket方式：

fastcgi_pass unix:/dev/shm/php-cgi.sock;

php-fpm.conf中配置链接监听：

tcp方式：

listen = 127.0.0.1:9000

socket方式：

listen = /dev/shm/php-cgi.sock

tcp方式：tcp协议监听本机9000端口，需要通过网络连接，占用端口，占用网络资源

socket方式：使用unix domain socket连接套接字/dev/shm/php-cgi.sock，不走网络，速度更快，/dev/shm是内存文件系统挂载点，比默认的/tmp/php-cgi.sock磁盘文件更快

修改完重启php-fpm和nginx（nginx -s reload）可以生效

注意：nginx和php-fpm需要配置一样的链接方式和监听方式，就是php-fpm使用什么监听，nginx就需要用什么方式来链接
