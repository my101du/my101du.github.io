系统自带

当您启动一个新的 Laravel 项目时，错误和异常处理就已为您配置就绪。应用程序触发的所有异常都被 App\Exceptions\Handler 类记录下来，然后渲染给用户

Laravel 使用 Monolog 库进行日志处理，它能为各种强大的日志处理程序提供支持。Laravel 配置了多几种日志处理程序，方便您在单个日志文件、多个交替日志文件之间进行选择写入，或将错误信息写入系统日志

配置

是否向用户显示错误信息

修改 config/app.php 文件的 debug 选项 或者 .env 文件中的 APP_DEBUG 环境变量。决定是否向用户显示错误信息

注意 本地开发环境下，应该将 APP_DEBUG 环境变量设置为 true 。在您的生产环境中，此值应始终保持 false 。如果你在生产中将该值设置为 true ，则可能会将敏感的配置值暴露给应用程序的最终用户。

存储

支持 single 、daily 、 syslog 和 errorlog 日志写入模式。

修改 config/app.php 配置文件中的 log 选项。例如按天来生成独立的日志文件

'log' => 'daily'

默认保留5天，如果要调整这个限制

'log_max_files' => 30

日志级别

默认存储全部日志级别，生产环境中应当记录的最低的日志级别

//默认的 log_level 被设置为 error 那么将会记录 error , critical , alert 和 emergency 日志信息：
'log_level' => env('APP_LOG_LEVEL', 'error'),

Monolog 识别以下严重级别 - 从低到高为: debug, info, notice, warning, error, critical, alert, emergency。

自定义 Monolog 日志

如果你想让你的应用程序完全控制 Monolog ，可以使用应用程序的 configureMonologUsing 方法。你应该放置一个回调方法到 bootstrap/app.php 文件中，在文件返回 $app 变量之前，调用这个方法：

$app->configureMonologUsing(function ($monolog) {
    $monolog->pushHandler(...);
});

return $app;

异常处理

所有异常都由 App\Exceptions\Handler 类处理。 这个类包含两个方法：

report render

Report 方法

report 方法用于记录异常或将其发送到外部服务，如 Bugsnag 或 Sentry。默认情况将异常传递给记录异常的基类

例如，如果您需要以不同的方式报告不同类型的异常，您可以使用 PHP instanceof 比较运算符

public function report(Exception $exception)
{
    if ($exception instanceof CustomException) {
        //
    }

    return parent::report($exception);
}

对应的辅助函数report()

报告一个异常，但又想继续处理当前的请求.report 辅助函数允许您使用异常 handler 的 report 方法快速报告一个异常而不抛出一个错误页面：

public function isValid($value)
{
    try {
        // 校验值...
    } catch (Exception $e) {
        report($e);

        return false;
    }
}

通过类型忽略异常

异常 handler 的 $dontReport 属性包含不会记录的异常类型数组。例如，404错误导致的异常以及其他几种类型的错误不会写入您的日志文件。您可以根据需要向此数组添加其他异常类型：

protected $dontReport = [
    \Illuminate\Auth\AuthenticationException::class,
    \Illuminate\Auth\Access\AuthorizationException::class,
    \Symfony\Component\HttpKernel\Exception\HttpException::class,
    \Illuminate\Database\Eloquent\ModelNotFoundException::class,
    \Illuminate\Validation\ValidationException::class,
];

Render 方法

render 方法负责将给定的异常转换成一个需要回传给浏览器的 HTTP 响应

默认情况下，异常会被传递给为您生成响应的基类。但是，您可以自由检查异常类型或返回您自己的自定义响应

public function render($request, Exception $exception)
{
    if ($exception instanceof CustomException) {
        return response()->view('errors.custom', [], 500);
    }

    return parent::render($request, $exception);
}

自定义异常的 report & render 方法

您并不一定要在异常 handler 中的 report 和 render 方法中处理异常，而是可以直接在自定义异常中定义 report 和 render 方法。如果这些方法存的在话，框架会自动调用他们：

namespace App\Exceptions;

use Exception;

class RenderException extends Exception
{
    /**
     * 报告异常
     *
     * @return void
     */
    public function report()
    {
        //
    }

    /**
     * 报告异常
     *
     * @param  \Illuminate\Http\Request
     * @return void
     */
    public function render($request)
    {
        return response(...);
    }
}

HTTP 异常

一些异常描述了来自服务器的 HTTP 错误代码。例如这可能是「找不到页面」 错误（404），「未授权错误」（401）甚至是由开发者造成的 500 错误。

可以使用 abort 辅助函数，在应用程序中的任何地方生成这样的响应（以及文本） 将立即创建一个被异常 handler 渲染过的异常.

abort(404);
abort(403, 'Unauthorized action.');
abort(503, 'Server Error.');

然后修改对应的视图文件resources/views/errors/503.blade.php

把错误消息显示出来{{ $exception->getMessage() }}（显示上面的 Server Error .）

 <div class="content">
    <div class="title">Error: {{ $exception->getMessage() }}</div>

</div>

自定义 HTTP 错误页面

例如，如果您要自定义 404 HTTP 状态代码的错误页面，请创建一个 resources/views/errors/404.blade.php

此文件将会用于渲染您应用中产生的所有 404 错误。此目录中的视图文件命名应与它们对应的 HTTP 状态代码匹配

由 abort 函数引发的 HttpException 实例将作为 $exception 变量传递给视图。

<h2>{{ $exception->getMessage() }}</h2>

记录：使用 Log Facade 将信息写入日志

Laravel 在强大的 Monolog 库上提供了一个简单的抽象层。默认情况下，Laravel 日志文件的存储目录被配置为 storage/logs

可以使用 Log facade 将信息写入日志

//...
use Illuminate\Support\Facades\Log;
//..

Log::info('Showing user profile for user: '.$id);

除了 info,可以按级别来记录

  • Log::emergency($message);
  • Log::alert($message);
  • Log::critical($message);
  • Log::error($message);
  • Log::warning($message);
  • Log::notice($message);
  • Log::info($message);
  • Log::debug($message);

记录上下文信息

以数组格式传递给日志方法，将被格式化并与日志消息一起显示

Log::info('User failed to login.', ['id' => $user->id]);

访问底层 Monolog 实例

多种额外的 handler 用来记录日志

$monolog = Log::getMonolog();

通过 Event 来实现

Model 的 create update 等动作中都会产生一些事件，通过监听这些事件，然后进行记录。

使用库 Laravel ActivityLog

当然更方便地是使用一些第三方的包

https://github.com/spatie/laravel-activitylog https://docs.spatie.be/laravel-activitylog/v1/introduction

清理日志

你可以使用 artisan 命令来清理超过多少天的日志,

php artisan activitylog:clean

这个时间在 config 文件有配置, 名称delete_records_older_than_days

或者使用 Scheduler 来实现

//app/Console/Kernel.php
protected function schedule(Schedule $schedule)
{
   $schedule->command('activitylog:clean')->daily();
}

记录特定 Model 的 events

除了自己手动指定 subject (发生事件的主体，一个模型)，还可以让 Model 在发生一些事件的时候 自动记录日志(例如创建, 修改, 删除删除数据，就和我们前面在模型发生 updated, created 等消息时手动记录日志信息一样)

让模型能在事件发生时自动记录日志

只需要让模型使用Spatie\Activitylog\Traits\LogsActivity这个 trait

use Illuminate\Database\Eloquent\Model;
use Spatie\Activitylog\Traits\LogsActivity;

class NewsItem extends Model
{
    use LogsActivity;
    //...

然后在模型类上设置$logAttributes, 记录该模型哪些属性的变化会记录日志.

use Illuminate\Database\Eloquent\Model;
use Spatie\Activitylog\Traits\LogsActivity;

class NewsItem extends Model
{
    use LogsActivity;

    protected $fillable = ['name', 'text'];
    
    protected static $logAttributes = ['name', 'text'];
}

那么当创建一个模型时, 一个 $activity 实例会记录哪些内容?

$newsItem = NewsItem::create([
   'name' => 'original name',
   'text' => 'Lorum'
]);

//creating the newsItem will cause an activity being logged
$activity = Activity::all()->last();

$activity->description; //returns 'created'
$activity->subject; //returns the instance of NewsItem that was created
$activity->changes; //returns ['attributes' => ['name' => 'original name', 'text' => 'Lorum']];

以 User 模型为例

然后看看在更新的时候, 会记录哪些?

$newsItem->name = 'updated name'
$newsItem->save();

//updating the newsItem will cause an activity being logged
$activity = Activity::all()->last();

$activity->description; //returns 'updated'
$activity->subject; //returns the instance of NewsItem that was created

$activity->changes;
//return this array
/*
[
   'attributes' => [
        'name' => 'updated name',
        'text' => 'Lorum',
    ],
    'old' => [
        'name' => 'original name',
        'text' => 'Lorum',
    ],
];
*/

description 字段的值是updated

自定义记录的事件

默认会记录 created updated deleted 类型的事件, 如果不需要全部记录, 只需要记录某种事件. 通过设置$recordEvents 成员可以实现。

use Illuminate\Database\Eloquent\Model;
use Spatie\Activitylog\Traits\LogsActivity;

class NewsItem extends Model
{
    use LogsActivity;

    //only the `deleted` event will get logged automatically
    protected static $recordEvents = ['deleted'];
}

自定义 description 的内容

默认情况下 description 只会记录几个简单的字符串 如 created, updated, deleted. 如果想让它看起来更易读, 可以重写getDescriptionForEvent()方法

use Illuminate\Database\Eloquent\Model;
use Spatie\Activitylog\Traits\LogsActivity;

class NewsItem extends Model
{
    use LogsActivity;

    protected $fillable = ['name', 'text'];

    public function getDescriptionForEvent(string $eventName): string
    {
        return "This model has been {$eventName}";
    }

}

再次修改 model 后, 记录的 activity 中，description 字段的内容就变成了

‘This model has been created’

忽略特定属性的记录

如果想让 model 的某些属性不被记录, 可以定义 $ignoreChangedAttributes 的值

如下, 首先通过logAttributes记录’name’, ‘text’两个属性的变化, 然后通过ignoreChangedAttributes忽略 text 的 change action 记录.(会记录 created deleted ?)

use Illuminate\Database\Eloquent\Model;
use Spatie\Activitylog\Traits\LogsActivity;

class NewsItem extends Model
{
    use LogsActivity;
    
    protected static $ignoreChangedAttributes = ['text'];

    protected $fillable = ['name', 'text'];
    
    protected static $logAttributes = ['name', 'text'];
}

只记录被改变的属性

如果定义了$logAttributes成员，里面定义了要记录的所有字段。 同时，希望只记录“actually changed after the update”的属性，可以使用$logOnlyDirty

Changing only name means only the name attribute will be logged in the activity, and text will be left out.

(这段不甚明白，是不是说一个模型的某个属性值被改变，但没有真实 update 到数据表，就不会记录？ 要通过代码测试)

use Illuminate\Database\Eloquent\Model;
use Spatie\Activitylog\Traits\LogsActivity;

class NewsItem extends Model
{
    use LogsActivity;

    protected $fillable = ['name', 'text'];
    
    protected static $logAttributes = ['name', 'text'];
    
    protected static $logOnlyDirty = true;
}

使用 CausesActivity 这个 trait，让 Model 能得到属于自己的 activities

为了得到满足条件的 activities 记录，可以使用 Eloquent 的方法。

例如前面我们已经知道的

$activityResult = Activity::all()->last();

或者取回作用在某个 Model 上的 activities（“subject_type”字段记录）

$activityResult = Activity::where('subject_type', 'App\User')->get();

这样的方式是以“Activity”模型来修改查询条件得到结果的。

我们还可以直接让 Model 来查询属于它的 activity（自动筛选）

通过向 Model 添加 获取 activities 的方法，它提供了一个activity的“关联”(relationship)，会返回所有作用在 Model 上的 activities

例如，在一个 User 模型中包含它，则会返回当前用户的 activities

修改 \App\User.php 这个模型

//...
use Spatie\Activitylog\Traits\CausesActivity;

class User extends Model{
	use CausesActivity;
	//....

然后在 Controller 某个方法中

public function getLogs(){
	dd(\Auth::user()->activity);
}

就会把所有当前登录用户触发的 activities 全部返回来

问题

使用这个特性后，会导致原来的 use Spatie\Activitylog\Traits\LogsActivity;（让 Model 自动记录 activity） 和它产生 activity冲突。。。两者不能同时用。。。。

FatalErrorException in User.php line 17: Trait method activity has not been applied, because there are collisions with other trait methods on App\User

禁用、启用 logging

如果不想对指定的 Model 进行 log，则可以禁用

同样，可以对已经禁用了 logging 的 Model 重新启用日志记录

$newsItem = NewsItem::create([
   'name' => 'original name',
   'text' => 'Lorum'
]);

// Updating with logging disabled
$newsItem->disableLogging();

$newsItem->update(['name' => 'The new name is not logged']);

// Updating with logging enabled
$newsItem->enableLogging();

$newsItem->update(['name' => 'The new name is logged']);

使用占位符

在记录 activity 的时候，可以使用占位符 :subject, :causer or :properties 来自动获取和替换成实际的数据。例如：

activity()
    ->performedOn($article)
    ->causedBy($user)
    ->withProperties(['laravel' => 'awesome'])
    ->log('The subject name is :subject.name, the causer name is :causer.name and Laravel is :properties.laravel');

多个 log

默认 log name

如果不特别指定log_name, 则把所有 activities 保存到"default” 这个默认 log name

这个“default”名字可以在配置文件中修改default_log_name实现

activity()->log('hi');

$lastActivity = Spatie\Activitylog\Models\Activity::all()->last();

$lastActivity->log_name; //returns 'default';

在写入 activity 的时候指定 log name

传递一个参数给activity()来指定 log name

activity('other-log')->log("hi");

Activity::all()->last()->log_name; //returns 'other-log';

为不同的 Model 指定一个 log name

默认情况下， LogsActivity traits 使用配置文件中定义的 default_log_name， 可以通过重写 Model 的 getLogNameToUse函数来指定 log name

public function getLogNameToUse(string $eventName = ''): string
{
   return 'custom_log_name_for_this_model';
}

取回指定 log 的 activitys

实际上 Activity 就是一个 Eloquent Model，因此要获取一个 log name 下的所有满足条件的 activities，也可以使用 Eloquent 的方法

Activity::where('log_name' , 'other-log')->get(); //returns all activity from the 'other-log'

或者用一个inLog方法来指定查找 activities 的作用域（scope？是不是就是 log name）

Activity::inLog('other-log')->get();

//you can pass multiple log names to the scope
Activity::inLog('default', 'other-log')->get();

//passing an array is just as good
Activity::inLog(['default', 'other-log'])->get();
