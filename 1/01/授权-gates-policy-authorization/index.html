授权 Gates & Policy （Authorization）

slug: laravel-user-permission-control date: 2017/07/01 10:10:10 tags:

  • laravel

用户正常通过认证后（用户认证服务 登录/注册等），不可能让每一个进入系统的用户都拥有一样的操作许可，必须按需求划分许可等级。

Laravel 用两种方式实现用户授权：

gates 许可？ 和 policy 策略？

gates 可以看做 路由器，提供了一个简单的、基于闭包的方式来授权认证 policy 则像控制器，在特定的模型或者资源中通过分组来实现授权认证的逻辑

大部分应用同时包含 gates 和 policy。两者结合起来使用。

Gates 大部分应用在模型和资源无关的地方，比如查看管理员的面板。与之相反，策略应该用在特定的模型或者资源中。

Gates

定义

Gates 是用来决定用户是否授权执行给定的 动作的闭包函数

在app/Providers/AuthServiceProvider.php文件中，使用Gate facade 来定义许可。

Gates 总是默认接受一个用户实例作为第一个参数，并且可以接受可选参数，比如 相关的 Eloquent 模型。

定义一个update-post许可，判断当前 User 和 Post 模型是否有从属关系。（用户的 id 是否与文章的 user_id 一致）

public function boot(GateContract $gate)
    {
        $this->registerPolicies($gate);

        /*$gate->define('update-post', function ($user, $post) {
            return $user->id === $post->user_id;
        });*/
        
        //5.5文档不同
        Gate::define('update-post', function ($user, $post) {
            return $user->id == $post->user_id;
        });
    }

也可以使用 Class@method 风格的回调字符串来定义，就像路由里定义一个对应的控制器方法一样

注意，这里已经出现了 Policy 概念了，后面会讲到

public function boot()
{
    $this->registerPolicies();

    Gate::define('update-post', 'PostPolicy@update');
    Gate::define('posts.view', 'PostPolicy@view');
    Gate::define('posts.create', 'PostPolicy@create');
    Gate::define('posts.update', 'PostPolicy@update');
    Gate::define('posts.delete', 'PostPolicy@delete');
}

资源 Gates

一次性定义多个 Gate 功能，类似路由里定义资源

Gate::resource('posts', 'PostPolicy');

//等于下面几句
Gate::define('posts.view', 'PostPolicy@view');
Gate::define('posts.create', 'PostPolicy@create');
Gate::define('posts.update', 'PostPolicy@update');
Gate::define('posts.delete', 'PostPolicy@delete');

//如果需要覆盖或添加新功能， 传入第三个参数
//下面的代码将创建两个新的Gate定义： posts.image 和 posts.photo 
Gate::resource('posts', 'PostPolicy', [
    'image' => 'updateImage',
    'photo' => 'updatePhoto',
]);

授权检查前后

可以使用 before 方法来定义所有授权检查前会被运行的回调。例如下例，如果是管理员（before回调返回一个非 null 的结果），该结果会被作为检查的结果，将中断后面的其他验证

$gate->before(function ($user, $ability) {
    if ($user->isSuperAdmin()) {
        return true;
    }
});

既然有 before，那么就有 after。表示当所有授权检查后会被运行的回调，无法修改 after 回调中授权检查的结果

$gate->after(function ($user, $ability, $result, $arguments) {
    //
});

授权动作检查

定义完授权后，就可以使用来做“许可检查”了。

使用 gates 来授权动作时，应使用 allows 或 denies 方法

allows（别名 check）, denies。它们获取许可的名称和参数，并传递到许可的回调中。Gate 会自动加载当前登录用户，因此不需要传递当前登录用户到方法中。

在某个 Controller 中，在 update 方法中检查许可。当然你也可以用 allows 方法，不过判断结果就是反的了。

public function update($id)
    {
        $post = Post::findOrFail($id);

        if (Gate::allows('update-post', $post)) {
            // 指定用户可以更新博客...
        }

        if (Gate::denies('update-post', $post)) {
            abort(403);
        }

        // 更新文章...
    }

forUser($user)`处理 非当前登录用户 的其他指定用户

if (Gate::forUser($user)->allows('update-post', $post)) {
    //
}

许可的回调传递多个参数 ？？5.5没有了

这里不是用前面的 GateContract，而是 Gate Facade

Gate::define('delete-comment', function ($user, $post, $comment) {
    //
});

然后在做许可检查的时候，传入对应的数组作为 Gate 方法的参数

if (Gate::allows('delete-comment', [$post, $comment])) {
    //
}

User 模型直接使用 Authorizable trait，不需要单独用 Gate Facede

除了用 Gate 外，还可以用 App\User 模型的实例检查许可，User 模型使用了Authorizable trait，它提供了两个方法：can 及 cannot（非常像 allows 和 denies）

public function update(Request $request, $id)
{
    $post = Post::findOrFail($id);

    if ($request->user()->cannot('update-post', $post)) {
        abort(403);
    }

    // 更新文章...
}

在 Blade 模板中

有时候需要在模板中快速判断许可并决定显示的内容

@can('update-post', $post)
    <a href="/post/{{ $post->id }}/edit">编辑文章</a>
@else
    <!-- 当前登录用户不可以更新文章 -->
@endcan

在表单请求中

在表单请求的 authorize 方法中进行检查

public function authorize()
{
    $postId = $this->route('post');

    return Gate::allows('update', Post::findOrFail($postId));
}

策略 Policy

策略是在特定**模型或者资源****中组织授权逻辑的类 (Gate 一般指定某个动作——当然也有resource类型的 Gate)

例如，如果你的应用是一个博客，会有一个 Post 模型和一个相应的 PostPolicy 来授权用户动作，比如创建或者更新博客

生成策略

使用 make:policy artisan 命令 来生成策略，放在app/Policies目录

//默认生成空的策略类
php artisan make:policy PostPolicy

//如果希望生成的类包含基本的「CRUD」策略方法， 可以在使用命令时指定 --model 选项：
php artisan make:policy PostPolicy --model=Post

会生成文件类似

namespace App\Policies;

use App\User;
use Illuminate\Auth\Access\HandlesAuthorization;

class UserPolicy
{
    use HandlesAuthorization;

    /**
     * Determine whether the user can view the user.
     *
     * @param  \App\User  $user
     * @param  \App\User  $user
     * @return mixed
     */
    public function view(User $user, User $user)
    {
        //
    }

    public function create(User $user)
    {
        //
    }
    
    //other code

所有授权策略会通过 Laravel 服务容器 解析，意指你可以在授权策略的构造器对任何需要的依赖使用类型提示，它们将会被自动注入

注册策略

在 AuthServiceProvider 包含了一个 policies 属性，可将各种 模型(如 Post::class) 对应至 管理它们的授权策略 (PostPolicy::class)

/**
 * 应用程序的授权策略对应。
 *
 * @var array
 */
protected $policies = [
    Post::class => PostPolicy::class,
    
    // 最新版 laravel 上面的这个，需要 use 准确的类路径  否则会出现即使条件满足，也返回“This action is unauthorized.”。
    
    //还可以用下面的“字符串”的方法，测试ok
    'App\Entities\Ticket' => 'App\Policies\TicketPolicy'
];

将全部的授权逻辑定义在 AuthServiceProvider 文件中，会越来越庞杂，因此在大型的应用中，需要把这些授权逻辑切分至「授权策略」类，基于授权的资源****将授权逻辑进行分组

编辑授权策略

一旦授权策略被生成且注册，我们就可以为授权的每个动作添加方法。

(你可以安装 spatie/laravel-permission 扩展包，来参考它的 UserPolicy 已有的一些方法)

在 PostPolicy 中定义一个 update 方法，它会判断指定的 User 是否可以更新指定的 Post 实例。接收 User 和 Post 实例作为参数，返回 true 或 false

注意看看，它和 Gates 里面定义的 update-post许可有什么相同和不同？

namespace App\Policies;

use App\User;
use App\Post;

class PostPolicy
{
    /**
     * 判断指定的文章是否可以被该用户更新。
     *
     * @param  \App\User  $user
     * @param  \App\Post  $post
     * @return bool
     */
    public function update(User $user, Post $post)
    {
        return $user->id === $post->user_id;
    }
}

你可以接着在此授权策略定义额外的方法，作为各种许可需要的授权。例如，你可以定义 show、destroy 或 addComment 方法来授权 Post 的多种行为。

如果在 Artisan 控制台生成策略时使用 –model 选项，会自动包含view、create、update 和 delete 动作。

注意：所有授权策略会通过 Laravel 服务容器 解析，意指你可以在授权策略的构造器对任何需要的依赖使用类型提示，它们将会被自动注入。

策略过滤器

对特定用户，你可能希望通过指定的策略授权所有动作（这个功能最常见的场景是授权应用的管理员可以访问所有动作）。

和前面在 AuthServiceProvider 中定义 Gate 一样，可以在授权策略赋予所有许可给指定用户，在授权策略中定义一个 before 方法。授权策略的此方法会在其它所有授权检查前被运行，提供了一种方式来授权动作而不是指定的策略方法来执行判断。

public function before($user, $ability)
{
    if ($user->isSuperAdmin()) {
        return true;
    }
}

如果你想拒绝用户所有的授权，你应该在 before 方法中返回 false。如果返回的是 null，则通过其它的策略方法来决定授权与否。

策略方法没有关联的模型

注意到下面的代码，Policy 类中，update方法有两个参数，第二个是 Post 模型，而create只有一个参数。

可见一些策略方法只接受当前认证通过的用户作为参数而不用传入授权相关的模型实例（因为还没创建模型之前，User 和 Model 之间并没有关联），常见的就是 create 动作。

public function create(User $user)
{
    //
}

public function update(User $user, Post $post)
{
    //
}

例如我们只需要检查下用户是否有权创建博客，而不需要知道“博客模型的实例”（因为还没创建呢）,只需要传入一个参数就可以了。

使用 Policy 来授权动作

通过用户模型

User 模型的 can 与 cannot 方法。

如果指定模型的 Policy 已被注册，can 方法会自动调用核实的策略方法并且返回 boolean 值。

如果没有策略注册到这个模型，can 方法会尝试调用和动作名相匹配的基于闭包的 Gate。

策略 Policy，优先于许可 Gate

if ($user->can('update', $post)) {
    //
}

前面提到有一种授权策略不需要模型实例（例如用户能否 create 博客），则在 can等判断里也不需要传入模型实例，只要传入类名

use App\Post;

if ($user->can('create', Post::class)) {
    // 执行相关策略中的「create」方法...
}

通过中间件

Laravel 包含一个可以在请求到达路由或控制器之前就进行动作授权的中间件。默认Illuminate\Auth\Middleware\Authorize 中间件被指定到 App\Http\Kernel 类中 can 键上

use App\Post;

Route::put('/post/{post}', function (Post $post) {
    // 当前用户可以更新博客...
})->middleware('can:update,post');

传递给 can 中间件 2 个参数,第一个是需要授权的动作的名称(update),第二个是我们希望传递给策略方法的路由参数(post, 这里因为使用了 隐式模型绑定-routing.html#implicit-binding，一个 Post 会被传递给策略方法)

如果用户不被授权访问指定的动作，这个中间件会生成带有 403 状态码的 HTTP 响应。

同样对于不需要指定模型的策略，can 动作也可以只需要指定类名

Route::post('/post', function () {
    // 当前用户可以创建博客...
})->middleware('can:create,App\Post');

通过控制器辅助函数

除了在 User 模型中提供辅助方法外，Laravel 也为所有继承了 App\Http\Controllers\Controller 基类的控制器提供了一个有用的 authorize 方法

和 can 方法类似，这个方法接收需要授权的动作和相关的模型作为参数。如果动作不被授权，authorize 方法会抛出 Illuminate\Auth\Access\AuthorizationException 异常，然后被 Laravel 默认的异常处理器转化为带有 403 状态码的 HTTP 响应：

class PostController extends Controller
{
    /**
     * 更新指定博客。
     *
     * @param  Request  $request
     * @param  Post  $post
     * @return Response
     */
    public function update(Request $request, Post $post)
    {
        $this->authorize('update', $post);

        // 当前用户可以更新博客

同样。。。不需要指定模型的动作的

$this->authorize('create', Post::class);

使用 Blade 模板

希望页面的指定部分只展示给允许授权访问指定动作的用户.

例如，你可能希望只展示更新表单给有权更新博客的用户。这种情况下，你可以直接使用 @can 和 @cannot 指令。

@can('update', $post)
    <!-- 当前用户可以更新博客 -->
@elsecan('create', $post)
    <!-- 当前用户可以新建博客 -->
@endcan

@cannot('update', $post)
    <!-- 当前用户不可以更新博客 -->
@elsecannot('create', $post)
    <!-- 当前用户不可以新建博客 -->
@endcannot

实际上是 @if @unless 的缩写, @can 和@cannot会转换成下面的声明

@if (Auth::user()->can('update', $post))
    <!-- 当前用户可以更新博客 -->
@endif

@unless (Auth::user()->can('update', $post))
    <!-- 当前用户不可以更新博客 -->
@endunless

又来。。。不需要指定模型实例的动作

@can('create', App\Post::class)
    <!-- 当前用户可以新建博客 -->
@endcan

@cannot('create', App\Post::class)
    <!-- 当前用户不可以新建博客 -->
@endcannot
