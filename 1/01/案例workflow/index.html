案例：workflow

开发

前面几个资料讲了 workflow 的组成部分，以及数据表设计、workflow、workflow nodes、ticket、ticket log 等类设计。

这里用 Symfony 的 WorkFlow 引擎来实现

选择

Laravel-Workflow : 有局限

把工作流数据保存到数据库

查看这个文件 https://github.com/brexis/laravel-workflow/blob/master/src/WorkflowRegistry.php

public function __constructor() {
    //...
    foreach ($this->config as $name => $workflowData) {
        $this->addFromArray($name, $workflowData);
    }
}

发现是只能从配置文件config/workflow.php里，读取工作流的配置数据。 添加、编辑一个单独的文件不太灵活；当数据增多，查看一个文件里多个条目就很容易出错了（重名等问题）。

dump 工作流为dot->图片

另外，好像它的 dump 只能在 command 里跑， 不能直接用 php 代码来调用。

phpmentors-jp/workflower ：太复杂

https://github.com/phpmentors-jp/workflower/blob/master/docs/quick-start-guide.md

文档太复杂了。但这个图可以参考（joget 里，也可以定义这样的，在不同的“角色”之间流动，角色内部也有transactions）, 还有不同的 node/condition switchs(start, true, false, end)

还有设计思想“Managing business processes” 仔细阅读

因此用 symfony workflow component

symfony workflow component 更灵活一些，可以实现我们

  1. 使用数据库来保存 workflow 数据
  2. dump 命令可以方便地在代码里调用
  3. 使用起来比较简单

表/模型与代码里的逻辑

前面有一些案例，都是自己写表结构、模型、字段来实现工作流、节点等等，灵活性较大。

但因为这里要用 Symfony 的工作流引擎，我的思路是：对应 yaml 文件的不同节点来设计数据表，模型尽可能体现 places, transitions，marking 几个概念，把其他的 角色、条件什么的附加到模型的方法上去。

方法并不是最佳的，后面逐步完善和优化。

分析

通过一个实例来剖析下一个工作流的组成部分：

底层的数据模型

  • 工作流有基本信息： 例如“请假申请工作流”

  • 工作流有对应的表单：（为简化,就1个）用户要填一些字段信息， 1)请假人 2)请假部门 3)请多久 4)什么原因等。

  • 工作流有N个节点(places)： 0)开始 1)等待经理审核 2)经理已通过 3)经理已拒绝 4）等待HR审核 5)HR已通过 6)HR已拒绝 7)结束

    • 节点有一些访问权限： 这里 节点0)可以任何雇员访问 节点1)等待经理审核 只有该员工的部门经理可访问。经理想看自己还有哪些 ticket 没处理完的，首先筛选自己有权限访问的 节点，然后把节点对应 tickets 数据列出来。
    • 节点动作: 触发跳转 transition

  • 每一个节点有多个跳转(transition)： 在节点1)上就有两个可能的跳转动作，a)经理通过 b)经理拒绝，分别跳到 节点3)和节点4)

    • 跳转条件，满足条件，才能跳转到下一个节点
      • 人工触发 (节点上有权限操作的人点击 审核/通过 按钮，然后)
      • 自动(例如定时任务，到时间自动打回； 或者判断当申请人的角色是部门经理，自动进入下一个节点)
    • 触发动作
      • 例如请假申请被通过/拒绝，发邮件给申请人，然后自动备案

（为简化，先都用人工审核，即使部门经理自己发布请假信息，也要自己通过一下，后面改成自动）

完成上面这些底层的组成部分后，下面来分析具体的”申请单“

  • 用户基于某个工作流+表单，创建一个申请单(ticket->model)： 例如编号 “QJ20180101”

  • 申请单的 state 字段(对应工作流yaml的marking)表示当前处于工作流的哪个节点，并列出该节点所有对应的 transitions。

  • 有权限的用户(部门经理看到本部门员工的请假申请)，可以修改它当前所在的 state (apply), 例如从 1)等待经理审核 –> 2)经理已通过.

    • 不管是手动，还是自动，在进入、完成一个 transition 的时候都可以触发动作 （runTransitionEvents）

  • 其他

    • 每个申请单的creator,通过它看到自己的单整个状态
    • 创建一个申请单后，默认位于 0)开始 节点。
    • 定时任务，或者每次修改、打开申请单，自动检查所有以 当前节点开始的“跳转条件”，是否能自动进入下一个节点， 并在下一个节点又再次检查“跳转条件”，一直到最后无法自动跳转，留在那里； 如果没有自动跳转，则等待人工审核

WorkFlow 工作流

默认属性

  • supports ： App\Entities\WorkFlowTicket
  • marking_store-> arguments: state ()
  • audit_trail->enabled: true
  • type: workflow

可修改属性

  • places: 即 workflow nodes，还可以给它赋予“条件”，满足后才能跳转到别的节点(next, prev)

WorkFlow Nodes 工作流节点 (place/stage….) 如何控制流动

在上面绑定各种事件，触发条件、类型、执行人等。根据业务自行调整

当工作流到了这个节点上，就需要审批人来触发它进行跳转

如果满足特定的条件，则跳到不同的下一个节点，例如

自动

（1）到了时间还没有人审核 （2）请假时间 < 1 天

人工

（3）自己是普通员工，则先发送到本部门经理审核，然后去人事部 （4）自己是经理，也要走”发送到部门经理“，但是是”自动通过“，然后去人事部

动作触发

（1）审核通过 （2）审核拒绝

ticket 每次变动一次 ，就检查一下 conditions, 看能否自动/手动跳到下一个节点。 (apply)

WorkFlowTransitions 节点过渡(以及条件)

WorkFlowForm 表单与字段

WorkFlowTicket 审批项目 (model, has the “status”)

WorkFlowTicketLogs 审批项目日志 ()

其他

WorkFlowEvents （暂时不需要，目前也就发邮件、短信两种事件，可以写死。将来可能服务器相关会有更多）
