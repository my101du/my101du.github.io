灵活的数据分类联动选择

以招聘信息为例

terms [数据条目 的 参数值列表]

按参数 1 “地区"有 北京市|朝阳区|西城区(有的选市范围, 有的选区范围); 按参数 2 “招聘种类"属于 社会招聘|校园招聘|全部(有的招聘信息不会分社会/校园，有的则需要细分) 那么可以把这些参数”值“统一都放到一张表里面去， 1: 朝阳区; 2: 西城区; 3: 北京市; 4: 社会招聘; 5: 校园招聘; 6: 全部 …

注意，这里没有体现出 “父级”关系，仅仅保存一个参数值的”清单“

CREATE TABLE `wp_terms` (
  `term_id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
  `name` varchar(200) COLLATE utf8mb4_unicode_520_ci NOT NULL DEFAULT '',
  `slug` varchar(200) COLLATE utf8mb4_unicode_520_ci NOT NULL DEFAULT '',
  `term_group` bigint(10) NOT NULL DEFAULT '0',
  PRIMARY KEY (`term_id`),
  KEY `slug` (`slug`(191)),
  KEY `name` (`name`(191))
) ENGINE=MyISAM AUTO_INCREMENT=261 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_520_ci;

term_taxonomy [参数值列表 的 参数类别与层级]

对上面的参数值列表： (1)按照参数 进行标注(归类) (2)表明父级关系

  1. term_id=1 (朝阳区), taxonomy=地区, parent=3, count=1
  2. term_id=2 (西城区), taxonomy=地区, partent=3, count=2
  3. term_id=3 (北京市), taxonomy=地区, parent=0, count=10
  4. term_id=4 (社会招聘), taxonomy=招聘种类, parent=6, count=4
  5. term_id=5 (校园招聘), taxonomy=招聘种类, parent=6, count=5
  6. term_id=6 (全部), taxonomy=招聘种类, parent=0, count=20

注意 子类的 count 加起来并 不等于 父类的 count 父类与子类，仅仅是方便逻辑上的理解，实际上子类的数据并不会"放入"父类，统计时是独立的。

这个表的 taxonomy 可以包括各种”类型“，例如针对 product 的 product_category, product_type. 针对文章的 category, post_tag(对， tag 也是一种分类的参数)

这里 partent 是具体某个参数值的层级，而不是自身的层级，例如 category 和 post_tag 是平级的，并没层级关系，只是一个分类方法

UNIQUE KEY term_id_taxonomy(term_id, taxonomy)

这个 KEY 干啥的。。。。普通索引。

CREATE TABLE `wp_term_taxonomy` (
  `term_taxonomy_id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
  `term_id` bigint(20) unsigned NOT NULL DEFAULT '0',
  `taxonomy` varchar(32) COLLATE utf8mb4_unicode_520_ci NOT NULL DEFAULT '',
  `description` longtext COLLATE utf8mb4_unicode_520_ci NOT NULL,
  `parent` bigint(20) unsigned NOT NULL DEFAULT '0',
  `count` bigint(20) NOT NULL DEFAULT '0',
  PRIMARY KEY (`term_taxonomy_id`),
  UNIQUE KEY `term_id_taxonomy` (`term_id`,`taxonomy`),
  KEY `taxonomy` (`taxonomy`)
) ENGINE=MyISAM AUTO_INCREMENT=261 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_520_ci;

考虑

如果我们的项目不需要单独对 terms 数据进行维护，完全可以直接把 terms 里面的 name/slug 直接放到 terms_taxonomy 里替换 term_id 不需要维护一个单独的表。

termmeta

CREATE TABLE `wp_termmeta` (
  `meta_id` bigint(20) unsigned NOT NULL AUTO_INCREMENT,
  `term_id` bigint(20) unsigned NOT NULL DEFAULT '0',
  `meta_key` varchar(255) COLLATE utf8mb4_unicode_520_ci DEFAULT NULL,
  `meta_value` longtext COLLATE utf8mb4_unicode_520_ci,
  PRIMARY KEY (`meta_id`),
  KEY `term_id` (`term_id`),
  KEY `meta_key` (`meta_key`(191))
) ENGINE=MyISAM AUTO_INCREMENT=348 DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_520_ci;

term_relationships [数据条目 的参数信息]

通过结合上面两个表，能够得到得到参数值的信息（参数名、层级），现在需要和某条数据关联起来 比如在发布一个招聘信息的时候，同时勾上"北京市"和"朝阳区”.那么会得到

  1. job_id=1, term_taxonomy_id=3, order=1
  2. job_id=1, term_taxonomy_id=2, order=2
  3. job_id=1, term_taxonomy_id=6, order=1
  4. job_id=1, term_taxonomy_id=4, order=2

可见一条招聘信息，有多少个参数，在这个表里面就会存在多少条记录 每个参数值，还有附加的 order, 排列顺序

最后就是 jobs 表了（略）对应上面 term_relationships 的 job_id
